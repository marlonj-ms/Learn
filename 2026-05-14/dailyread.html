1. Lambda 表达式（Lambda Expression）

Lambda 是一段匿名（没有方法名）的方法行为。

number => number * 10

意思：

输入 number  →  返回 number * 10

等价的普通方法：

int TimesTen(int number)
{
    return number * 10;
}
关键认知：
  • Lambda 本身没有独立的类型。
  • 它必须被一个委托类型（delegate type）承载，才能存在并被调用。
  • 承载它的"容器"可以是：
    1. 一个显式的委托变量（如 Func<int,int> x = ...）
    2. 一个方法参数的类型（如 Where 的 Func<T,bool> 参数）

2. 委托（Delegate）的本质

委托（delegate） = 一个能"装住方法"的类型，可以把方法当变量传递、存储、调用。

int Triple(int number)
{
    return number * 3;
}

Func<int, int> operation = Triple;   // 把方法装进委托变量
Console.WriteLine(operation(5));      // 调用，输出 15

内存示意：

operation ──> (delegate instance) ──> Triple 方法的代码
委托存的是什么？
  • ✅ 一个指向方法的引用
  • ❌ 不是方法执行的结果
  • ❌ 不是方法代码的副本

类比：operation 像遥控器，它不装电视，但能控制电视。

3. Func / Action / Predicate

这三个是 C# 内置的常用泛型委托类型。

委托类型用途返回值形状示例
Func<T, TResult> 有返回值的方法 有 int → int
Action<T> 无返回值的方法 void string → void
Predicate<T> 判断真假 bool T → bool 
Func<int, int>     timesTen   = number => number * 10;
Func<string, int>  getLength  = name   => name.Length;
Func<int, bool>    isEven     = number => number % 2 == 0;
Action<string>     log        = message => Console.WriteLine(message);
Predicate<string>  startsA    = name => name.StartsWith("A");

4. 方法组（Method Group） vs 调用方法

这是最容易混淆的点之一。

int AddOne(int number)
{
    return number + 1;
}

Func<int, int> a = AddOne;       // ✅ 传入方法本身（method group）
Func<int, int> b = AddOne(5);    // ❌ 编译错误：把 int 6 赋给 Func
写法含义
AddOne方法本身（一段行为）
AddOne(5)调用方法，得到结果 int 6

LINQ 中的同类陷阱：

numbers.Select(TimesTen);       // ✅ 把方法传给 Select
numbers.Select(TimesTen(5));    // ❌ 传的是 int 50，不是方法

5. 自定义 delegate 关键字

除了 Func / Action / Predicate，C# 还允许自定义委托类型，给委托一个有意义的名字。

public delegate decimal DiscountRule(decimal originalPrice);

decimal ApplyDiscount(decimal price, DiscountRule rule)
{
    return rule(price);
}

DiscountRule tenPercentOff = price => price * 0.9m;
DiscountRule flatTwentyOff = price => price - 20m;

Console.WriteLine(ApplyDiscount(100m, tenPercentOff));  // 90
Console.WriteLine(ApplyDiscount(100m, flatTwentyOff));  // 80
什么时候用 Func，什么时候用自定义 delegate？
  • 99% 情况使用 Func / Action / Predicate
  • 当"名字本身能解释意图"时，用自定义 delegate（如 DiscountRule、ValidationRule、Logger）
  • 事件（event）几乎总是基于自定义 delegate

6. LINQ 与 Lambda / Delegate 的关系

容易出错的认知：
"LINQ 和 lambda 无关" ❌ 错。
LINQ 几乎完全依赖委托和 lambda。

Where 的真实签名（简化版）：

IEnumerable<T> Where<T>(
    this IEnumerable<T> source,
    Func<T, bool> predicate)        // ← 参数类型是委托类型

Select 的真实签名（简化版）：

IEnumerable<TResult> Select<T, TResult>(
    this IEnumerable<T> source,
    Func<T, TResult> selector)      // ← 参数类型是委托类型

所以当你写：

numbers.Where(n => n > 3);

编译器在做的事：

1. 看到 lambda：n => n > 3
2. 看到 Where 的第二参数类型是 Func<int, bool>
3. 把 lambda 编译成 Func<int, bool> 委托实例
4. 把这个委托实例传给 Where
5. Where 循环 numbers，对每个元素调用委托，根据 bool 决定保留与否

整体流程图：

你写：    n => n * 10               (lambda 行为)
              │
              ▼
存入：    Func<int, int>            (委托类型)
              │
              ▼
传给：    numbers.Select(...)        (LINQ 方法)
              │
              ▼
得到：    IEnumerable<int>           (新集合)

7. Where vs Select 的形状

方法用途Lambda 形状返回什么
Where 过滤（filter） T → bool 原元素中满足条件的子集
Select 转换（transform） T → 任意类型 转换后的新元素 
// Where 必须返回 bool
numbers.Where(n => n > 3);          // ✅
numbers.Where(n => n + 1);          // ❌ 返回 int，不是 bool

// Select 接受任意返回类型
numbers.Select(n => n * 10);        // int -> int    ✅
numbers.Select(n => n.ToString());  // int -> string ✅
numbers.Select(n => n > 3);         // int -> bool   ✅（合法但少见）
记忆口诀：
  • Where 在问："我要不要这个元素？"（答案必须是 true/false）
  • Select 在问："这个元素要变成什么？"（答案可以是任何东西）

8. 为什么委托设计强大（Open/Closed Principle）

对比两种写死和可扩展的设计：

❌ 写死的版本

void DoWork()
{
    Console.WriteLine("Starting work");
    // ... 工作内容 ...
    Console.WriteLine("Work done");
}

问题：如果想换成文件日志、数据库日志、单元测试中静默，都得修改 DoWork 本身。

✅ 委托注入的版本

public delegate void Logger(string message);

void DoWork(Logger log)
{
    log("Starting work");
    // ... 工作内容 ...
    log("Work done");
}

Logger consoleLogger = m => Console.WriteLine($"[CONSOLE] {m}");
Logger fileLogger    = m => File.AppendAllText("log.txt", m + "\n");
Logger silentLogger  = m => { };  // 单元测试用

DoWork(consoleLogger);
DoWork(fileLogger);
DoWork(silentLogger);
开闭原则（Open/Closed Principle, OCP）：
  • 对扩展开放：随时添加新的 Logger 实现
  • 对修改关闭：从此不需要再改 DoWork

这是 SOLID 原则中的 "O"，也是依赖注入、策略模式（Strategy Pattern）的基础。

验证规则的可组合性

public delegate bool ValidationRule(string input);

ValidationRule notEmpty    = text => !string.IsNullOrWhiteSpace(text);
ValidationRule isEmail     = text => text.Contains("@");
ValidationRule biggerThan8 = text => text.Length >= 8;

// 组合规则
ValidationRule emailAndLongEnough = text =>
    notEmpty(text) && isEmail(text) && biggerThan8(text);

这就是生产环境中 FluentValidation 等库的底层思路。

9. 常见陷阱与 var 规则

写法结果原因
var x = n => n + 1; ❌ 编译错误 var 无法从 lambda 推断类型，lambda 本身没有类型
Func<int,int> x = n => n + 1; ✅ 合法 显式委托类型承载 lambda
var c = numbers.Where(n => n > 1); ✅ 合法 var 推断的是 Where 的返回值（IEnumerable<int>），不是 lambda
numbers.Where(n => n + 1); ❌ 编译错误 Where 需要返回 bool，但 lambda 返回 int
numbers.Select(TimesTen(5)); ❌ 编译错误 传入的是结果 int，不是方法
口诀： var 不能装 lambda，但可以装"调用结果"。

10. 小测练习与答案

每天看完后建议蒙住答案，自测一遍。

练习

// 行 1
Func<int, int> a = n => n + 1;

// 行 2
var b = n => n + 1;

// 行 3
List<int> numbers = new() { 1, 2, 3 };
numbers.Where(n => n > 1);

// 行 4
var c = numbers.Where(n => n > 1);

// 行 5
Func<int, bool> rule = n => n > 1;
numbers.Where(rule);

// 行 6
numbers.Where(n => n + 1);

答案

行结果原因
1✅显式委托类型，形状 int → int 匹配
2❌var 不能从 lambda 推断类型
3✅Where 期望 Func<int,bool>，n > 1 返回 bool
4✅var 推断的是 Where 的返回值 IEnumerable<int>
5✅先装进委托变量再传给 Where，完全合法
6❌n + 1 返回 int，不是 bool；这种用法应该给 Select

11. Console.WriteLine 集合陷阱

下面这段代码不会报错，但输出可能让你怀疑人生：

List<int> numbers = new() { 1, 2, 3 };
var result = numbers.Where(x => x > 1);
Console.WriteLine(result);

实际输出：

System.Linq.Enumerable+WhereListIterator`1[System.Int32]
为什么不报错？
  • Console.WriteLine 有一个重载接受 object?，任何类型都能传入
  • 编译合法，运行不抛异常
  • 但它会调用对象的 ToString()，而 LINQ 迭代器没有重写 ToString()
  • 所以打印出来的是类型全名，不是集合内容

正确打印集合内容的 3 种方式

// 方式 1：foreach（最显式）
foreach (int n in result)
{
    Console.WriteLine(n);
}

// 方式 2：string.Join（最常用）
Console.WriteLine(string.Join(", ", result));   // 输出: 2, 3

// 方式 3：先 ToList 再观察（调试常用）
List<int> materialized = result.ToList();
口诀： 集合不能直接 WriteLine，要么 foreach，要么 string.Join。

12. 延迟执行（Deferred Execution）

LINQ 链式调用本身不执行任何工作。它只是建立一份"配方"，等"消费者"触发时才工作。

List<int> numbers = new() { 1, 2, 3 };

var result = numbers.Select(n =>
{
    Console.WriteLine($"processing {n}");
    return n * 10;
});

Console.WriteLine("--- before iteration ---");

foreach (int x in result)
{
    Console.WriteLine($"got {x}");
}

输出（注意顺序）：

--- before iteration ---
processing 1
got 10
processing 2
got 20
processing 3
got 30
关键认知：
  • var result = numbers.Select(...) 这一行什么都没做
  • 只是建立"配方"，记住"想做什么"
  • 触发执行的"消费者"包括：foreach / ToList() / ToArray() / Count() / First() / Sum()

延迟执行的真实陷阱

List<int> numbers = new() { 1, 2, 3 };
var result = numbers.Where(x => x > 1);   // 配方，不执行

numbers.Add(99);                            // 修改源

foreach (int n in result)                   // 现在才执行
{
    Console.WriteLine(n);
}
// 输出：2, 3, 99    ← 99 也被纳入了！

因为 result 是在 foreach 时才"现去 numbers 拿数据过滤"，那时 99 已经在里面了。

13. 流式 vs 非流式（Streaming vs Buffering）

流式（Streaming）— 一次拉一个

大部分 LINQ 方法是流式的：消费者要一个，它才处理一个。

List<int> numbers = new() { 1, 2, 3 };
var streamed = numbers.Select(n => 
{
    Console.WriteLine($"select {n}");
    return n * 10;
});

foreach (int x in streamed) Console.WriteLine($"got {x}");

输出：

select 1
got 10
select 2
got 20
select 3
got 30

非流式（Buffering）— 一次性吃完所有元素

OrderBy / GroupBy / Reverse 必须先看完所有元素才能给出第一个结果。

var buffered = numbers.OrderByDescending(n => 
{
    Console.WriteLine($"order {n}");
    return n;
});

foreach (int x in buffered) Console.WriteLine($"got {x}");

输出：

order 1
order 2
order 3        ← 所有元素都先被处理
got 3
got 2
got 1          ← 然后才开始输出

分类速查

方法类型说明
Where / Select流式每拉一个处理一个
Take / Skip流式按需拉取，达到数量停止
TakeWhile / SkipWhile流式条件满足时停止/开始
OrderBy / OrderByDescending非流式必须看完所有元素才能排序
GroupBy非流式必须看完所有元素才能分组
Reverse非流式必须看完所有元素才能反转
ToList / ToArray非流式立即触发并物化所有元素
Count / Sum / Min / Max非流式必须遍历所有元素
First / FirstOrDefault流式找到第一个就停止
Any / All流式可以提前退出

14. Take 位置决定上游做多少无用功

三种写法，都是"过滤 + 转换 + 取 3 个"，但效率和正确性差距巨大。

// 写法 1：Where → Select → Take    ✅ 最优
bigList.Where(n => n > 10).Select(n => Expensive(n)).Take(3);

// 写法 2：Select → Where → Take    ⚠️ 浪费
bigList.Select(n => Expensive(n)).Where(n => n > 10).Take(3);

// 写法 3：Select → Take → Where    ❌ 语义错误
bigList.Select(n => Expensive(n)).Take(3).Where(n => n > 10);
写法Expensive 调用次数结果元素数评价
1. Where → Select → Take33✅ 最优
2. Select → Where → Take6+（取决于过滤率）3⚠️ 浪费
3. Select → Take → Where30（可能为空）❌ 错误
黄金顺序： 
源 → Where（过滤）
   → Select（转换）
   → OrderBy / GroupBy（如需要）
   → Take / Skip / First（截断）

口诀：先筛掉，再加工，再排序，最后截取。

Take 在前面的陷阱

List<int> numbers = new() { 5, 3, 8, 1, 9 };

var x = numbers.Take(3).Where(n => n > 4);
// Take(3) 先拿 {5, 3, 8}，再过滤 → {5, 8}

var y = numbers.Where(n => n > 4).Take(3);
// 先过滤出 {5, 8, 9}，再 Take(3) → {5, 8, 9}

同样的方法，不同顺序，结果完全不同。

15. 方法参数立即求值 vs Lambda 延迟执行

这是一个非常容易混淆的点：不是所有写在 LINQ 链里的东西都是延迟的。

Func<int, int> takewithmessage = n =>
{
    Console.WriteLine("one time"); 
    return n;
};

var result = bigList
    .Take(takewithmessage(3))    // ⚠️ 这里立即执行
    .Where(n => n > 10);

Console.WriteLine("--- before iteration ---");
foreach (int x in result) { ... }

输出：

one time                       ← 在 var result = ... 这一行就打印了
--- before iteration ---
（foreach 输出...）
为什么 "one time" 没有延迟？
  • takewithmessage(3) 不是 lambda，是对方法的调用
  • C# 规则：调用一个方法时，参数会先被求值，然后才传进去
  • 所以 Take(takewithmessage(3)) 实际上：
    1. 立即调用 takewithmessage(3) → 打印 "one time"，返回 3
    2. 然后调用 Take(3)，返回一个迭代器（这部分是延迟的）

对比清单

写法何时执行
Take(takewithmessage(3))立即（参数求值）
Take(SomeMethod())立即（参数求值）
Take(Math.Max(1, 2))立即（参数求值）
Where(n => SomeMethod(n))延迟（lambda 体）
Select(n => Expensive(n))延迟（lambda 体）
口诀：
  • LINQ 方法的参数会立即求值
  • LINQ 方法接收 lambda 之后，lambda 体才延迟执行

另一个微妙对比

// A：RandomNumber 只调用 1 次（在 .Take() 这一行）
var a = bigList.Take(RandomNumber());

// B：RandomNumber 在每次 Where 检查时都调用 1 次
var b = bigList.Where(n => n < RandomNumber());

16. 上游执行次数 ≥ 下游执行次数

在流式 LINQ 链中，越靠左（上游）的操作，lambda 调用次数越多。

var result = bigList
    .Where(n => n > 10)                  // 上游
    .Select(n => Expensive(n))           // 中游
    .Take(3);                            // 下游

foreach (int x in result) { ... }

对于 bigList = 1..1_000_000，调用次数：

操作lambda 调用次数说明
Where13检查 1~13 才凑够 3 个 > 10 的
Select (Expensive)3只对通过 Where 的元素跑
Take 输出3到 3 后整条链路停止
规律： 
Where 调用次数  ≥  Select 调用次数  =  Take 输出次数

因为下游每要 1 个元素，上游可能要"消化"多个被过滤掉的元素才能给出 1 个通过的。

设计启示：
  • ✅ 把便宜的过滤放上游（Where 在 Select 前）
  • ✅ 把昂贵的转换放下游（Select 接近末端）
  • ✅ 把截断放最末（Take / First）

17. 为什么要传方法而不是直接调用？

一个核心问题：既然方法内部可以直接调用别的方法，为什么还要把方法作为参数传进去？

一句话答案：

"传方法" ≠ "调用方法"，而是"暂不决定具体调用什么，留给调用者决定"。 这就是控制反转（Inversion of Control, IoC）的核心思想。

对比：写死 vs 留洞

维度直接调用委托传递
灵活性行为写死行为可替换
测试性难以单元测试容易注入假实现
扩展性改需求要改源代码加新行为不动旧代码
复用性一个方法只能干一件事一个方法服务多种用法
设计紧耦合（tight coupling）松耦合（loose coupling）

场景 1：同一段逻辑，多种用法

// ❌ 硬编码
void PlaceOrder(int orderId)
{
    Console.WriteLine($"Placing order {orderId}");
    // ...
}

// ✅ 行为外部注入
void PlaceOrder(int orderId, Action<string> log)
{
    log($"Placing order {orderId}");
    // ...
}

// 调用方决定怎么记日志
PlaceOrder(123, msg => Console.WriteLine(msg));               // 开发
PlaceOrder(123, msg => File.AppendAllText("log.txt", msg));   // 测试
PlaceOrder(123, msg => Telemetry.Track(msg));                 // 生产
PlaceOrder(123, msg => { });                                   // 单元测试

场景 2：单元测试不可绕过

// ❌ 真实 API 写死，单元测试会真的扣钱
void ProcessPayment(int amount)
{
    bool success = CreditCardApi.Charge(amount);
}

// ✅ "如何收费"作为参数传入
void ProcessPayment(int amount, Func<int, bool> chargeMethod)
{
    bool success = chargeMethod(amount);
}

// 生产
ProcessPayment(100, amount => CreditCardApi.Charge(amount));

// 测试：假装成功 / 失败，不动真实接口
ProcessPayment(100, amount => true);
ProcessPayment(100, amount => false);

这就是依赖注入（dependency injection）的核心思想。

场景 3：写库给别人用

想象你是 LINQ 的作者，写 Where 方法：

// ❌ 无法预测用户的过滤需求
IEnumerable<int> Where(IEnumerable<int> source)
{
    foreach (int n in source)
        if (n > 10) yield return n;   // 写死过滤规则
}

// ✅ 让用户传入过滤规则
IEnumerable<T> Where<T>(IEnumerable<T> source, Func<T, bool> predicate)
{
    foreach (T item in source)
        if (predicate(item)) yield return item;
}

一个 Where 服务所有过滤需求 → LINQ 才能存在。

场景 4：行为在运行时决定（开闭原则）

// ❌ if-else 写死，加新角色就要改这里
void OnButtonClick()
{
    if (userRole == "admin") OpenAdminPanel();
    else if (userRole == "user") OpenUserPanel();
    else if (userRole == "guest") ShowLoginDialog();
}

// ✅ 行为存到字典中，按 key 查找
Dictionary<string, Action> roleHandlers = new()
{
    ["admin"] = () => OpenAdminPanel(),
    ["user"]  = () => OpenUserPanel(),
    ["guest"] = () => ShowLoginDialog(),
};

void OnButtonClick()
{
    roleHandlers[userRole]();
}

// 加新角色：只需加一行，不动 OnButtonClick
roleHandlers["vip"] = () => OpenVipPanel();

场景 5：事件 / 回调 / 异步

button.Click += () => Console.WriteLine("clicked");
//                       ↑ 委托。没有它，UI 框架不存在。

httpClient.GetAsync(url).ContinueWith(task => ...);
//                                       ↑ 异步回调。也是委托。

timer.Elapsed += (sender, args) => DoSomethingPeriodic();
//                                       ↑ 事件。也是委托。
整个 .NET 框架都建立在委托之上。 没有委托：没有事件、没有异步回调、没有 LINQ、没有依赖注入、没有 ASP.NET 的请求处理管道。

比喻：DVD 机和电影

直接调用 = 一台只能播 1 部固定电影的 DVD 机
           （内部硬编码电影，永远只能看这部）

委托传递 = 普通 DVD 机 + 可替换碟片
           （DVD 机不关心碟片内容，碟片可以无限替换）

委托设计就是把"电影"和"DVD 机"解耦。

18. 核心一句话总结

  1. Lambda = 匿名方法行为，必须有委托类型承载
  2. Delegate = 能装住方法的类型，让方法变成可传递的值
  3. Func / Action / Predicate = 最常用的内置泛型委托
  4. LINQ = 高阶方法 + 你传入的委托（lambda）来定义规则
  5. 价值 = 解耦行为与流程，实现 Open/Closed，便于扩展和测试
  6. 延迟执行 = LINQ 链是"配方"，消费者触发时才工作
  7. 流式处理 = Where / Select 拉一个处理一个；OrderBy / GroupBy 必须先吃完
  8. 顺序很重要 = 先 Where 再 Select 再 Take，效率最高
  9. 参数立即求值 = Take(M()) 中 M() 立即调用，只有 lambda 体才延迟
  10. 上游 ≥ 下游 = 把便宜的过滤放前面，昂贵的转换放后面
