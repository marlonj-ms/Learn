1. 变量声明与 var

// 显式类型
int    age   = 30;
string name  = "Alice";
bool   isOn  = true;

// 用 var 让编译器推断类型（需要等号右边能推断出类型）
var age2  = 30;          // 推断为 int
var name2 = "Alice";     // 推断为 string
var list  = new List<int>();   // 推断为 List<int>
var 不能用的场景：
  • 没有初始化：var x; ❌
  • 初始化为 null：var x = null; ❌
  • 用来装 lambda：var f = n => n + 1; ❌（lambda 自己没有类型）
  • 字段或方法参数（var 只能用在方法内的局部变量）

2. 数组（Array）的多种定义

// 方式 1：显式 new（最完整写法）
int[] a = new int[] { 1, 2, 3 };

// 方式 2：省略 new，仅在变量声明时合法
int[] b = { 1, 2, 3 };

// 方式 3：collection expression（C# 12，最简洁）
int[] c = [1, 2, 3];

// 方式 4：先指定大小，后赋值
int[] d = new int[5];      // [0, 0, 0, 0, 0]
d[0] = 10;
d[1] = 20;

// 方式 5：用 var
var e = new int[] { 1, 2, 3 };    // ✅ 推断 int[]
var f = new[] { 1, 2, 3 };        // ✅ 推断 int[]（元素类型从值推断）
var g = { 1, 2, 3 };              // ❌ 不行，{...} 不是表达式

多维数组

// 二维数组
int[,] grid = new int[3, 4];        // 3 行 4 列
int[,] m = { { 1, 2 }, { 3, 4 } };  // 2x2

// 数组的数组（jagged array）
int[][] jagged = new int[3][];
jagged[0] = new int[] { 1, 2 };
jagged[1] = new int[] { 3, 4, 5 };

字符串数组的常见写法

string[] names1 = new string[] { "Alice", "Bob", "Charlie" };
string[] names2 = { "Alice", "Bob", "Charlie" };
string[] names3 = ["Alice", "Bob", "Charlie"];
常见错误：
  • 把 'a'（char）当 "a"（string）写：单引号是字符，双引号是字符串
  • 把 { 1, 2, 3 } 直接传给方法 —— 这种字面量只能用在变量声明右侧

3. List<T> 的定义与初始化

// 方式 1：空 List
List<int> a = new List<int>();

// 方式 2：target-typed new（C# 9，最常用）

names.Add("Charlie");           // 追加
names.AddRange(new[] { "D", "E" });  // 追加多个
names.Insert(0, "First");       // 在索引 0 处插入
names.Remove("Bob");            // 删除（第一个匹配）
names.RemoveAt(0);              // 按索引删除
names.Count;                    // 元素个数
names[0];                       // 索引访问
names.Contains("Alice");        // 是否存在
names.Sort();                   // 原地排序

4. Dictionary<K,V> 的定义与初始化

// 方式 1：空字典
Dictionary<string, int> a = new Dictionary<string, int>();

// 方式 2：target-typed new
Dictionary<string, int> b = new();

// 方式 3：集合初始化器（用 { key, value } 形式）
Dictionary<string, int> c = new()
{
    { "alice", 30 },
    { "bob", 25 },
};

// 方式 4：索引器初始化器（推荐，更直观）
Dictionary<string, int> d = new()
{
    ["alice"] = 30,
    ["bob"]   = 25,
};

常用操作

Dictionary<string, int> ages = new()
{
    ["alice"] = 30,
    ["bob"]   = 25,
};

ages["charlie"] = 40;            // 添加或更新
ages.Add("david", 50);           // 添加（key 已存在则抛异常）
ages.Remove("bob");              // 删除
ages.ContainsKey("alice");       // 检查 key
ages["alice"];                   // 取值（key 不存在抛异常）

// 安全取值
if (ages.TryGetValue("alice", out int age))
{
    Console.WriteLine(age);
}

// 遍历
foreach (var kvp in ages)
{
    Console.WriteLine($"{kvp.Key} = {kvp.Value}");
}

8. Lambda 表达式的各种形态

// 无参数
Action a = () => Console.WriteLine("hi");

// 一个参数（括号可省略）
Func<int, int> b = n => n * 2;
Func<int, int> c = (n) => n * 2;       // 等价

// 多个参数（括号必须）
Func<int, int, int> d = (a, b) => a + b;

// 显式类型（少用）
Func<int, int> e = (int n) => n * 2;

// 语句体（statement lambda）：多行逻辑用 { }
Func<int, int> f = n =>
{
    Console.WriteLine($"got {n}");
    return n * 2;
};

// 无返回值的 lambda（用于 Action）
Action<string> g = msg => Console.WriteLine(msg);
Action<string> h = msg =>
{
    Console.WriteLine("logging:");
    Console.WriteLine(msg);
};
表达式体 vs 语句体：
  • n => n * 2 = 表达式体（expression lambda），直接返回表达式的值
  • n => { return n * 2; } = 语句体（statement lambda），必须用 return

9. 字符串：插值、$、@、"""

string name = "Alice";
int age = 30;

// 拼接（最原始）
string s1 = "Name: " + name + ", Age: " + age;

// 插值字符串（推荐）
string s2 = $"Name: {name}, Age: {age}";

// 表达式可以直接放在 {}
string s3 = $"Total: {age * 2}";
string s4 = $"Upper: {name.ToUpper()}";

// 格式化
string s5 = $"Price: {123.456:F2}";   // F2 = 保留 2 位小数

// @ 字符串（不处理转义符）
string path1 = "C:\\Users\\Marlon";
string path2 = @"C:\Users\Marlon";    // 等价但更易读

// $@ 或 @$ 组合（插值 + 不转义）
string s6 = $@"Path: C:\Users\{name}";

// 多行字符串（C# 11+，用 """）
string s7 = """
    {
        "name": "Alice",
        "age": 30
    }
    """;

10. PascalCase / camelCase 命名约定

类型约定示例
类名（Class）PascalCasePerson, OrderService
方法（Method）PascalCaseGetName(), PlaceOrder()
属性（Property）PascalCaseName, OrderId
公共字段（public field）PascalCasepublic int Count;

局部变量（local var）camelCaseint orderCount
方法参数camelCasevoid M(int orderId)
私有字段（private field）_camelCaseprivate int _count;
接口I + PascalCaseIList, IDisposable
常见错误：
  • .where() ❌ → .Where() ✅（LINQ 方法都是 Pascal）

14. 类型转换（cast / convert）

// 隐式转换（安全，编译器自动）
int   i = 10;

double d = i;         // int -> double，OK

// 显式转换（cast，可能丢失精度或溢出）

// Convert 类（更通用）
int x1 = Convert.ToInt32("123");
double y1 = Convert.ToDouble("3.14");

// 引用类型转换
object o = "hello";
string s2 = (string)o;       // 失败抛异常
string s3 = o as string;     // 失败返回 null
if (o is string s4) { ... }  // 安全检查

15. foreach 与 for 的选择

List<string> names = new() { "Alice", "Bob", "Charlie" };

// foreach：遍历所有元素，最常用
foreach (string name in names)
{
    Console.WriteLine(name);
}

// for：需要索引时
for (int i = 0; i < names.Count; i++)
{
    Console.WriteLine($"{i}: {names[i]}");
}

// while：需要复杂退出条件
int j = 0;
while (j < names.Count && names[j] != "Bob")
{
    j++;
}

// do-while：至少执行一次
do
{
    // ...
} while (condition);
选择规则：
  • 大部分情况用 foreach，简单不易错
  • 需要修改集合（边遍历边删除）时不要用 foreach，会抛异常
  • 需要索引、需要倒序、需要跳元素 → 用 for
  • 条件不固定 → 用 while
↑ 回到顶部

16. 扩展方法（Extension Methods）

public static class MyExtensions   // ① 必须 static class
{
    public static bool IsEven(this int n)  // ② static  ③ this 第一参数
        => n % 2 == 0;

    // 多参数：第一参数自动从"点前"对象拿，后面的显式传
    public static int AddBy(this int age, int y) => age + y;
}

// 两种调用完全等价：
5.IsEven();                  // 点语法（漂亮）
MyExtensions.IsEven(5);      // 静态调用（编译器实际生成的）

10.AddBy(3);                 // → 13（this=10, y=3）
三铁律：
  1. 类必须是 static class
  2. 方法必须是 static
  3. 第一参数前加 this —— 决定能扩展给哪种类型
查找规则：编译器靠 using 命名空间 + this 参数类型定位扩展方法，跟 class 叫什么名字无关。 所以 using System.Linq; 才让 list.Where(...) 可用。
↑ 回到顶部

=> 三种身份对照

符号位置含义例子
= 变量声明后 赋值 int x = 5;
=> 参数列表后 n => Lambda 箭头（分隔参数和函数体） n => n + 1
=> 方法/属性签名后 ) => 表达式体（替代 { return ... }） int Add(int n) => n + 1; 
// 经典：一行里两个 => 含义完全不同
public Func<int, int> MakeAdder(int n) => x => x + n;
//                                     ↑   ↑
//                          表达式体（return）  Lambda 箭头
识别口诀：
  • 左边是参数 → Lambda 箭头
  • 左边是方法/属性签名 → 表达式体语法糖
陷阱：表达式体 => 后面只能跟一个表达式，不能跟代码块 { ... }。 多行逻辑必须老老实实写 { }。

18. 集合长度三表（Length / Count / Count()）

类型怎么拿长度类型
int[]（数组） arr.Length 属性
string s.Length 属性
List<T> list.Count 属性
Dictionary<K,V> / HashSet<T> .Count 属性
IEnumerable<T> seq.Count() LINQ 扩展方法
陷阱：IEnumerable<T> 没有 .Length 也没有 .Count 属性！ 要么用 LINQ 的 .Count()（会遍历整个序列），要么自己 foreach 计数。 原因：IEnumerable 可能是无限序列，没有"长度"的概念。 
// ❌ 错误：IEnumerable 没有 Length
public int Average2(IEnumerable<int> nums)
{
    int sum = 0;
    foreach (int n in nums) sum += n;
    return sum / nums.Length;   // 编译错误！
}

// ✅ 正确：边遍历边计数
public int Average2(IEnumerable<int> nums)
{
    int sum = 0, count = 0;
    foreach (int n in nums) { sum += n; count++; }
    return sum / count;
}

// ✅ 也可以用 LINQ
using System.Linq;
int avg = nums.Sum() / nums.Count();

19. LINQ 链路：管道思维 + 常用方法

方法作用lambda 形状
Where过滤T → bool
Select投影/变换T → TOut
OrderBy / OrderByDescending排序（不破坏原 list）T → TKey
First / FirstOrDefault取第一个可带 T → bool
Any / All断言T → bool
Distinct去重（无）
GroupBy按 key 分组T → TKey
Sum / Max / Min / Count聚合可选 T → 数值
Aggregate通用折叠(acc, T) → acc
LINQ 链不放副作用！不要在 Select 里 Console.WriteLine。它们是延迟执行（deferred execution）的数据管道，foreach 才真正触发。打印放外层 foreach 里。 
// ✅ 标准链式 + 打印分离
foreach (var t in booklist.Where(b => b.Pages > 500).Select(b => b.Title))
{
    Console.WriteLine(t);
}

// ✅ GroupBy 标准用法
foreach (var g in booklist.GroupBy(b => b.Author))
{
    var titles = string.Join(", ", g.Select(b => b.Title));
    Console.WriteLine($"{g.Key} ({g.Count()} books): {titles}");
}

// ✅ Aggregate 通用聚合
int total = booklist.Aggregate(0, (acc, b) => acc + b.Pages);
//                            ↑   ↑
//                         种子  累加规则

20. List.Sort + Comparison<T>

booklist.Sort((a, b) => a.Pages - b.Pages);   // 升序
booklist.Sort((a, b) => b.Pages - a.Pages);   // 降序
返回值含义
负数a 排在 b 前
0相等
正数b 排在 a 前
常见错误：写成 (a, b) => a.X < b.X（返回 bool）。Comparison<T> 要求返回 int，编译器报错。
OrderBy vs Sort：
list.OrderBy(...) — LINQ 扩展方法，返回新序列，原 list 不变。
list.Sort(...) — List 实例方法，原地排序，原 list 顺序被改。
溢出陷阱：页数很大时 a.X - b.X 可能溢出。生产代码用 a.X.CompareTo(b.X) 更稳。

21. Dictionary 计数模式（TryGetValue）

var counts = new Dictionary<string, int>();
foreach (var word in words)
{
    counts.TryGetValue(word, out int current);   // 不在时 current = 0
    counts[word] = current + 1;                  // 在则改不在则插
}
关键 trick：key 不存在时 out int current 被赋为 int 的默认值 0，所以省掉 if/else。
TryGetValue 只读，不会修改字典。真正插入的是后面那行 dict[k] = v。

三种"更新或插入"的对比

写法key 不存在key 已存在哈希查找次数
dict.Add(k, v)✅ 插入❌ 抛 ArgumentException1
dict[k] = v✅ 插入✅ 覆盖1
dict.TryAdd(k, v)✅ 插入，返回 true❌ 不改，返回 false1
if (ContainsKey(k)) ... else ...分支处理分支处理2
TryGetValue + 索引器赋值用默认值 + 插读出值 + 改1
 
// 索引器对 int 值的简写（在则改，不在则插入新 entry 值=1）
if (counts.ContainsKey(w))
    counts[w]++;            // 等价 counts[w] = counts[w] + 1
else
    counts.Add(w, 1);

// 解构遍历（KeyValuePair 支持解构）
foreach (var (key, value) in dict)
{
    Console.WriteLine($"{key}:{value}");

22. 反复踩的坑（持续累积）

1. 私有字段只能类内访问 
// 在 Main 里
var b = new Book(...);
Console.WriteLine(b._title);   // ❌ 编译错误，私有
Console.WriteLine(b.Title);    // ✅ 走 public 属性
2. 插值字符串少了 $ 
Console.WriteLine("{x}");      // ❌ 打印字面 {x}
Console.WriteLine($"{x}");     // ✅ 替换为 x 的值
3. string 是 immutable，方法返回新串 
name.Join(...);                // ❌ 不会修改 name
name = string.Join(", ", arr); // ✅ string.Join 是静态方法，要赋值
4. lambda 参数名遮蔽外层变量 
foreach (var g in groups)
{
    string.Join(",", g.Select(g => g.Title));   // 内层 g 是 Book，遮蔽外层 IGrouping
}
// ✅ 改名避免歧义
string.Join(",", g.Select(b => b.Title));
5. 不能 foreach 一边删元素 
foreach (var x in list)
{
    if (cond) list.Remove(x);   // ❌ 运行时 InvalidOperationException
}
// ✅ 用 RemoveAll 或 反向 for 循环
list.RemoveAll(x => cond);
6. 'word' 是 char 字面量，不是 string 
dict['word']    // ❌ 单引号里只能一个字符
dict["word"]    // ✅ 字符串字面量
dict[word]      // ✅ 变量（无引号）
7. EventHandler 和 EventHandler<T> 不要混用 
public event EventHandler? Rang;                    // e 是 EventArgs
public void OnRang(object? sender, DoorbellEventArgs e) { } // ❌ 签名不匹配

// ✅ 三处 T 必须一致
public event EventHandler<DoorbellEventArgs>? Rang;
public void OnRang(object? sender, DoorbellEventArgs e) { }
8. 想让 override 生效，父类方法必须调用 virtual 方法 
public void Press(string name)
{
  Rang?.Invoke(this, new DoorbellEventArgs(name)); // ❌ 绕过 OnRang，子类 override 不会执行
}

public void Press(string name)
{
  OnRang(new DoorbellEventArgs(name));             // ✅ 走 virtual dispatch
}
9. subscriber 不是 publisher 的子类 
public class Buzzer : TemperatureSensor { }  // ❌ Buzzer 不是一种 Sensor

public class Buzzer                          // ✅ Buzzer 是监听者
{
  public void OnThresholdExceeded(object? sender, ThresholdExceededEventArgs e) { }
}
10. var x = null; 编译不过 
var captured = null;                              // ❌ null 没类型，编译器推不出
ThresholdExceededEventArgs? captured = null;      // ✅ 显式声明可空类型
11. 测试方法不能重载（xUnit1024） 
// 产品代码这样写合法，搬进测试类就报错
[Theory] void Add_Works(int a, int b) { ... }           // ❌ xUnit1024
[Theory] void Add_Works(double a, double b) { ... }

// ✅ 改名让每个测试名唯一
[Theory] void Add_Works_Int(int a, int b) { ... }
[Theory] void Add_Works_Double(double a, double b) { ... }
原因：xUnit 的 Roslyn 分析器报错（不是 C# 编译器），CI 用方法名作测试唯一 ID，重名撞车。
12. Assert.Throws<T> 是精确类型匹配 
// 产品代码抛 ArgumentNullException（是 ArgumentException 的子类）
Assert.Throws<ArgumentException>(...);       // ❌ FAIL，不精确匹配
Assert.Throws<ArgumentNullException>(...);   // ✅ 精确匹配
Assert.ThrowsAny<ArgumentException>(...);    // ✅ 接受子类
13. x == double.NaN 永远 false 
if (x == double.NaN) { ... }      // ❌ 永远不进，IEEE 754: NaN != NaN
if (double.IsNaN(x)) { ... }      // ✅ 唯一正确的 NaN 检查
14. => 双重身份 —— 看位置区分 
Func<int,int> f = x => x * 2;     // 值的位置 → Lambda（匿名方法）
public int X => 42;               // 成员声明位置 → 表达式体成员（{ return 42; }）
两个 => 长得一样，意思完全不同。看出现在等号右边还是成员定义里来区分。
15. Assert.Equal(expected, actual) 参数顺序铁律 
Assert.Equal(actual, expected);    // ❌ 反了
Assert.Equal(expected, actual);    // ✅ expected 永远在前
反了 xUnit 失败信息会误导："Expected: 100, Actual: 101"，但其实是你写错了顺序。
16. [MemberData] 引用必须 public static 
private static IEnumerable<object[]> Data => ...;     // ❌ 私有不行
public IEnumerable<object[]> Data => ...;             // ❌ 实例成员不行
public static IEnumerable<object[]> Data => ...;      // ✅
另外 new[] { 1, 2 } 会推断成 int[]；[MemberData] 要 object[]，必须显式 new object[] { ... }。
17. 本地变量 / 参数必须 camelCase 
void M(double sensortemp, bool shouldraise)    // ❌ 全小写拼接
void M(double sensorTemp, bool shouldRaise)    // ✅ camelCase

bool eventfired = false;                        // ❌
bool eventFired = false;                        // ✅
类 / 方法 / 属性是 PascalCase；字段（私有）是 _camelCase；常量是 PascalCase。
18. 多目标 csproj：标签必须复数 + 分号 
❌ <TargetFramework>net10.0,net8.0</TargetFramework>     // 逗号 + 单数
❌ <TargetFramework>net10.0;net8.0</TargetFramework>     // 分号对了，但还是单数
✅ <TargetFrameworks>net10.0;net8.0</TargetFrameworks>   // 复数 s + 分号
错任何一处都报 NETSDK1046。报错信息会直接说 "use the 'TargetFrameworks' property instead" —— SDK 报错往往就是答案，认真读。
19. dotnet pack 不填元数据 → 占位垃圾 
// 你没填 PackageId/Version/Authors/Description，pack 出来的 .nuspec 长这样：
<id>TemperatureSensor.Core</id>              // 取的项目名
<version>1.0.0</version>                     // 假设的
<authors>TemperatureSensor.Core</authors>    // 又取项目名
<description>Package Description</description>  // 字面占位字符串！
id / version / authors / description 是 4 个必填字段。dependencies 可选。如果你上传 nuget.org 之前没改这些占位，所有用户搜到的都是"Package Description"。
20. <Version> + <PackageReleaseNotes> + 注释是三方契约 
// 你把版本从 1.1.0 → 1.1.1（PATCH 表示纯 bug 修复）
<Version>1.1.1</Version>
<!-- ❌ 注释还停在旧版本 -->
<!-- 1.1.0 → MINOR bump 加了新功能 -->
<PackageReleaseNotes>1.1.0: Added NaN validation</PackageReleaseNotes>  // ❌ 也没改
SemVer：PATCH（修 bug）/ MINOR（加功能 backward-compatible）/ MAJOR（破坏性改动）。三处要同步改，否则将来翻 git log 一脸懵。
21. DI 注册了不映射 → 编译过，运行炸 
services.AddTransient<OrderService>();          // ✅ 注册了 OrderService
// ❌ 没注册 IInventory → FakeInventory 的映射

provider.GetRequiredService<OrderService>();   // 💥 InvalidOperationException:
                                                //    Unable to resolve service for type 'IInventory'
// ✅ 补上接口映射
services.AddTransient<IInventory, FakeInventory>();
锚句："Registration ≠ construction"。容器只在 resolve 时才发现缺失依赖。Lifetime 控制的是复用策略，不是调用次数。
22. Scoped 注入 Singleton — 隐形数据陈旧 
services.AddSingleton<CacheService>();
services.AddScoped<DbContext>();
// CacheService 构造期注入 DbContext → 永远持有第一次 scope 的对象
// → 数据陈旧 / 内存泄漏
ASP.NET Core 在 Development 环境自动开 ValidateScopes = true 会在启动期抓出这种错。scope ≈ 一个 HTTP request。
23. appsettings.json 层级分隔符是 :，环境变量用 __ 
// appsettings.json
{ "Sensor": { "Threshold": 30.0 } }

builder.Configuration.GetValue<double>("Sensor:Threshold");   // ✅ 冒号 是层级分隔
builder.Configuration.GetValue<double>("Sensor.Threshold");   // ❌ 点不是层级分隔

// 环境变量（cmd/PowerShell 不允许 :，必须双下划线）
$env:Sensor__Threshold = "50"                                 // ✅ 双下划线 = :
配置提供者链 later wins：appsettings.json → appsettings.{Env}.json → 环境变量 → CLI 参数。
24. Minimal API：路由 handler 内部 += 订阅事件 = 内存泄漏 
// ❌ 每次请求都新增一个 handler，多播委托不去重
app.MapPost("/readings", (Sensor sensor, ILogger<Program> logger, ReadingDto dto) =>
{
    sensor.ThresholdExceeded += (s, e) => logger.LogWarning(...);
    sensor.RecordReading(dto.Celsius);
    return Results.Ok();
});

// ✅ 在组合根（composition root）订阅一次，启动期就位
var sensor = app.Services.GetRequiredService<Sensor>();
sensor.ThresholdExceeded += (s, e) => startupLogger.LogWarning(...);
app.Run();
4 次超温请求 = 1+2+3+4 = 10 条 warn，handler 永不释放。多播委托按引用比较，即使 lambda 长得一样也不去重。
25. Program.cs 顶级语句 → 测试看不见 Program 类 
// Program.cs 末尾少了这行
public partial class Program { }   // ← 没它，下面这行编译都过不去
// ↓ 测试项目
public class Tests : IClassFixture<WebApplicationFactory<Program>> { }
//                                                  ^^^^^^^ CS0122: inaccessible
顶级语句默认把 Program 做成 internal。微软官方推荐用 public partial class Program { } 开"测试小窗户"。
26. 测错误路径不能用 EnsureSuccessStatusCode() 
// ❌ 反路径用 EnsureSuccessStatusCode：400 会 throw，让"我期望 400"的测试假失败
response.EnsureSuccessStatusCode();
Assert.Equal(HttpStatusCode.BadRequest, response.StatusCode);   // 永远走不到这

// ✅ 测错误路径永远直接断言 StatusCode
Assert.Equal(HttpStatusCode.BadRequest, response.StatusCode);
正路径用 EnsureSuccessStatusCode() 没问题；反路径必须直接断言 StatusCode。
27. Dockerfile 必须放在它要 COPY 的所有项目的共同父目录 
// ❌ Dockerfile 放在子项目里 → 看不见兄弟项目
2026-05-25/
├── TemperatureSensor.Api/
│   └── Dockerfile              ← COPY ../TemperatureSensor.Core/ 失败
└── TemperatureSensor.Core/

// ✅ Dockerfile 放在共同父目录（common ancestor）
2026-05-25/
├── Dockerfile                  ← 这一层
├── TemperatureSensor.Api/
└── TemperatureSensor.Core/
docker build . 的 . 是 build context 根；COPY 路径相对它。锚句：Dockerfile 必须看得见所有它要 COPY 的项目。
28. Docker 多阶段：ENV / WORKDIR / 安装的包都不跨 stage，只有 COPY --from 是桥 
FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build
WORKDIR /src
RUN apt-get install -y vim          # ❌ Stage 2 没有 vim
ENV BUILD_TIME=2026-05-26           # ❌ Stage 2 拿不到这个变量

FROM mcr.microsoft.com/dotnet/aspnet:9.0
WORKDIR /app
COPY --from=build /app/publish .    # ✅ 这是唯一的桥
ENV ASPNETCORE_URLS=http://+:8080   # Stage 2 自己重新设
"FROM 是 stage 分界线"：每个 FROM 启动一个新的临时容器。"Stage 2 没编译器" — 想跑 dotnet build 都没工具。
29. PowerShell 的 curl 是别名陷阱，必须用 curl.exe 
// ❌ PowerShell 把 curl alias 到 Invoke-WebRequest，-X/-d 都不认
curl -X POST http://localhost:8080/readings -d '{"celsius":25}'
// → Invoke-WebRequest : A parameter cannot be found that matches parameter name 'X'

// ✅ 显式调 Windows 自带的 curl.exe（Win10+ 自带）
curl.exe -i -X POST http://localhost:8080/readings `
  -H "Content-Type: application/json" `
  -d "{\"celsius\":25}"                          # ← JSON 内引号双重转义
PowerShell 单引号字符串不会展开转义，JSON body 给 curl.exe 要写成 "{\"celsius\":25}"。
