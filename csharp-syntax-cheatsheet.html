2. 数组（Array）的多种定义

// 方式 1：显式 new（最完整写法）
int[] a = new int[] { 1, 2, 3 };

// 方式 2：省略 new，仅在变量声明时合法

// 方式 4：先指定大小，后赋值
int[] d = new int[5];      // [0, 0, 0, 0, 0]
d[0] = 10;
d[1] = 20;

// 方式 5：用 var
var e = new int[] { 1, 2, 3 };    // ✅ 推断 int[]
var f = new[] { 1, 2, 3 };        // ✅ 推断 int[]（元素类型从值推断）
var g = { 1, 2, 3 };              // ❌ 不行，{...} 不是表达式

多维数组

// 二维数组

int[,] m = { { 1, 2 }, { 3, 4 } };  // 2x2

// 数组的数组（jagged array）
int[][] jagged = new int[3][];
jagged[0] = new int[] { 1, 2 };
jagged[1] = new int[] { 3, 4, 5 };

字符串数组的常见写法

8. Lambda 表达式的各种形态

// 无参数
Action a = () => Console.WriteLine("hi");

// 一个参数（括号可省略）
Func<int, int> b = n => n * 2;
Func<int, int> c = (n) => n * 2;       // 等价

// 多个参数（括号必须）
Func<int, int, int> d = (a, b) => a + b;

// 显式类型（少用）
Func<int, int> e = (int n) => n * 2;

// 语句体（statement lambda）：多行逻辑用 { }
Func<int, int> f = n =>
{
    Console.WriteLine($"got {n}");
    return n * 2;
};

// 无返回值的 lambda（用于 Action）
Action<string> g = msg => Console.WriteLine(msg);
Action<string> h = msg =>
{
    Console.WriteLine("logging:");
    Console.WriteLine(msg);
};
表达式体 vs 语句体：
  • n => n * 2 = 表达式体（expression lambda），直接返回表达式的值
  • n => { return n * 2; } = 语句体（statement lambda），必须用 return

9. 字符串：插值、$、@、"""

string name = "Alice";
int age = 30;

// 拼接（最原始）
string s1 = "Name: " + name + ", Age: " + age;

// 插值字符串（推荐）
string s2 = $"Name: {name}, Age: {age}";

// 表达式可以直接放在 {}
string s3 = $"Total: {age * 2}";
string s4 = $"Upper: {name.ToUpper()}";

// 格式化
string s5 = $"Price: {123.456:F2}";   // F2 = 保留 2 位小数

// @ 字符串（不处理转义符）
string path1 = "C:\\Users\\Marlon";
string path2 = @"C:\Users\Marlon";    // 等价但更易读

// $@ 或 @$ 组合（插值 + 不转义）
string s6 = $@"Path: C:\Users\{name}";

// 多行字符串（C# 11+，用 """）
string s7 = """
    {
        "name": "Alice",
        "age": 30
    }
    """;

15. foreach 与 for 的选择

List<string> names = new() { "Alice", "Bob", "Charlie" };

// foreach：遍历所有元素，最常用
foreach (string name in names)
{
    Console.WriteLine(name);
}

// for：需要索引时
for (int i = 0; i < names.Count; i++)
{
    Console.WriteLine($"{i}: {names[i]}");
}

// while：需要复杂退出条件
int j = 0;
while (j < names.Count && names[j] != "Bob")
{
    j++;
}

// do-while：至少执行一次
do
{
    // ...
} while (condition);
选择规则：
  • 大部分情况用 foreach，简单不易错
  • 需要修改集合（边遍历边删除）时不要用 foreach，会抛异常
  • 需要索引、需要倒序、需要跳元素 → 用 for
  • 条件不固定 → 用 while

16. 扩展方法（Extension Methods）

public static class MyExtensions   // ① 必须 static class
{
    public static bool IsEven(this int n)  // ② static  ③ this 第一参数
        => n % 2 == 0;

    // 多参数：第一参数自动从"点前"对象拿，后面的显式传
    public static int AddBy(this int age, int y) => age + y;
}

// 两种调用完全等价：
5.IsEven();                  // 点语法（漂亮）
MyExtensions.IsEven(5);      // 静态调用（编译器实际生成的）

10.AddBy(3);                 // → 13（this=10, y=3）
三铁律：
  1. 类必须是 static class
  2. 方法必须是 static
  3. 第一参数前加 this —— 决定能扩展给哪种类型
查找规则：编译器靠 using 命名空间 + this 参数类型定位扩展方法，跟 class 叫什么名字无关。 所以 using System.Linq; 才让 list.Where(...) 可用。

18. 集合长度三表（Length / Count / Count()）

类型怎么拿长度类型
int[]（数组） arr.Length 属性
string s.Length 属性
List<T> list.Count 属性
Dictionary<K,V> / HashSet<T> .Count 属性
IEnumerable<T> seq.Count() LINQ 扩展方法
陷阱：IEnumerable<T> 没有 .Length 也没有 .Count 属性！ 要么用 LINQ 的 .Count()（会遍历整个序列），要么自己 foreach 计数。 原因：IEnumerable 可能是无限序列，没有"长度"的概念。 
// ❌ 错误：IEnumerable 没有 Length
public int Average2(IEnumerable<int> nums)
{
    int sum = 0;
    foreach (int n in nums) sum += n;
    return sum / nums.Length;   // 编译错误！
}

// ✅ 正确：边遍历边计数
public int Average2(IEnumerable<int> nums)
{
    int sum = 0, count = 0;
    foreach (int n in nums) { sum += n; count++; }
    return sum / count;
}

// ✅ 也可以用 LINQ
using System.Linq;
int avg = nums.Sum() / nums.Count();

19. LINQ 链路：管道思维 + 常用方法

方法作用lambda 形状
Where过滤T → bool
Select投影/变换T → TOut
OrderBy / OrderByDescending排序（不破坏原 list）T → TKey
First / FirstOrDefault取第一个可带 T → bool
Any / All断言T → bool
Distinct去重（无）
GroupBy按 key 分组T → TKey
Sum / Max / Min / Count聚合可选 T → 数值
Aggregate通用折叠(acc, T) → acc
LINQ 链不放副作用！不要在 Select 里 Console.WriteLine。它们是延迟执行（deferred execution）的数据管道，foreach 才真正触发。打印放外层 foreach 里。 
// ✅ 标准链式 + 打印分离
foreach (var t in booklist.Where(b => b.Pages > 500).Select(b => b.Title))
{
    Console.WriteLine(t);
}

// ✅ GroupBy 标准用法
foreach (var g in booklist.GroupBy(b => b.Author))
{
    var titles = string.Join(", ", g.Select(b => b.Title));
    Console.WriteLine($"{g.Key} ({g.Count()} books): {titles}");
}

// ✅ Aggregate 通用聚合
int total = booklist.Aggregate(0, (acc, b) => acc + b.Pages);
//                            ↑   ↑
//                         种子  累加规则

20. List.Sort + Comparison<T>

booklist.Sort((a, b) => a.Pages - b.Pages);   // 升序
booklist.Sort((a, b) => b.Pages - a.Pages);   // 降序
返回值含义
负数a 排在 b 前
0相等
正数b 排在 a 前
常见错误：写成 (a, b) => a.X < b.X（返回 bool）。Comparison<T> 要求返回 int，编译器报错。
OrderBy vs Sort：
list.OrderBy(...) — LINQ 扩展方法，返回新序列，原 list 不变。
list.Sort(...) — List 实例方法，原地排序，原 list 顺序被改。
溢出陷阱：页数很大时 a.X - b.X 可能溢出。生产代码用 a.X.CompareTo(b.X) 更稳。

21. Dictionary 计数模式（TryGetValue）

var counts = new Dictionary<string, int>();
foreach (var word in words)
{
    counts.TryGetValue(word, out int current);   // 不在时 current = 0
    counts[word] = current + 1;                  // 在则改不在则插
}
关键 trick：key 不存在时 out int current 被赋为 int 的默认值 0，所以省掉 if/else。
TryGetValue 只读，不会修改字典。真正插入的是后面那行 dict[k] = v。

三种"更新或插入"的对比

写法key 不存在key 已存在哈希查找次数
dict.Add(k, v)✅ 插入❌ 抛 ArgumentException1
dict[k] = v✅ 插入✅ 覆盖1
dict.TryAdd(k, v)✅ 插入，返回 true❌ 不改，返回 false1
if (ContainsKey(k)) ... else ...分支处理分支处理2
TryGetValue + 索引器赋值用默认值 + 插读出值 + 改1

22. 反复踩的坑（持续累积）

1. 私有字段只能类内访问 
// 在 Main 里
var b = new Book(...);
Console.WriteLine(b._title);   // ❌ 编译错误，私有
Console.WriteLine(b.Title);    // ✅ 走 public 属性
2. 插值字符串少了 $ 
Console.WriteLine("{x}");      // ❌ 打印字面 {x}
Console.WriteLine($"{x}");     // ✅ 替换为 x 的值
3. string 是 immutable，方法返回新串 
name.Join(...);                // ❌ 不会修改 name
name = string.Join(", ", arr); // ✅ string.Join 是静态方法，要赋值
4. lambda 参数名遮蔽外层变量 
foreach (var g in groups)
{
    string.Join(",", g.Select(g => g.Title));   // 内层 g 是 Book，遮蔽外层 IGrouping
}
// ✅ 改名避免歧义
string.Join(",", g.Select(b => b.Title));
5. 不能 foreach 一边删元素

44. 接口 ≠ DI — 看到 interface 别立刻喊 DI

public class Repository<T> where T : class, IAuditable   // ← 这里只是泛型约束
{
    public void Add(T item) { /* ... */ }
}
// ❌ "看到 interface 就以为在用 DI"
// ✅ DI 的关键标志是 IServiceCollection / AddSingleton / 构造函数注入
// ✅ 这个文件没那些 → 和 DI 一点关系都没有
锚句：接口 = C# 语言契约；DI = 设计模式 + 容器框架。DI 必用接口，但反之不成立。
45. => 三身份 — 位置决定身份，别立刻当 lambda 
// (1) Lambda — 输入到输出
list.Where(x => x > 0)

// (2) 表达式属性体 (expression-bodied property) — 简写 getter
public int Count => _items.Count;

// (3) 表达式方法体 — 简写单返回式方法
public override string ToString() => $"Order(Id={Id})";
锚句："=> 长得一样但身份三种。看它前面是什么：参数列表→lambda；属性签名→属性体；方法签名→方法体。"
46. new 的 4 种语法 — () 何时可省 
new Order() { Id = "A-100", Amount = 88.8m }     // (1) 完整：ctor + 初始化器
new Order   { Id = "A-100", Amount = 88.8m }     // (2) 省略空括号（仅无参构造时）
Order o = new() { Id = "A-100", Amount = 88.8m } // (3) target-typed (C# 9+)
var list = new List<int> { 1, 2, 3 };            // (4) 集合初始化器
锚句：(2) 不是"括号被吞了"，而是 C# 允许在 { } 跟随时省略无参括号。背后依然走 ctor + 属性赋值。
47. required × 默认值 — 默认永远不会赢 
public required string Id { get; init; } = "DEFAULT";
//          ↑ 编译期门卫              ↑ 运行期默认值

new Order { /* Id 没传 */ }   // ❌ CS9035 — 编译失败
new Order { Id = "ABC" }      // ✅ 必须显式传

// 即使写了 = "DEFAULT"，调用方仍必须显式赋值
// 因为 required 在编译期就拦下了，根本走不到运行期的默认值
锚句：required = 编译期门卫；默认值 = 运行期值。编译失败永远走不到运行期。
48. new() 约束是关于"方法体"，不是关于 T 本身 
// ❌ 没 new() 约束，方法体内不能 new T()
public class Repo<T> where T : class
{
    public T CreateDefault() => new T();   // ❌ 编译错
}

// ✅ 加上 new() 约束才能 new T()
public class Repo<T> where T : class, new()
{
    public T CreateDefault() => new T();   // ✅
}
锚句：new() 约束存在的全部理由就是让你能在泛型代码里 new T()。没这个需求就不要加。
49. CS9040 — required 和 new() 互斥 
public class Order : IAuditable
{
    public required string Id { get; init; }   // ← required 强制传值
}

public class Repository<T> where T : class, IAuditable, new()
{
    public T CreateDefault() => new T();   // ❌ CS9040
    //  T = Order，Order 有 required 成员 → 不能无参 new
}
锚句：required 把"必须传值"写进类型签名；new() 承诺"我能无参构造"。两个承诺直接打架，编译器替你提前发现。
50. 工厂方法 (factory method) — generic 代码里非常常见 
// Dictionary.GetOrAdd 经典签名
public TValue GetOrAdd(TKey key, Func<TKey, TValue> valueFactory);
//                                ──────────────────────────────
//                                把"怎么造 V"作为参数传进来

// EF Core
DbContextOptions.UseSqlServer(connectionString);   // 内部用 factory delegate 延迟构造

// 你今天看到的简化版（被 CS9040 挡住而注释掉的）
public T CreateDefault() => new T();   // ← "我自己也能造 T" 的工厂能力
锚句：工厂方法回答"谁来 new、什么时候 new"。在 generic + DI 场景里频繁出现。
51. 泛型类型替换发生在编译期 — 不是运行期 cast 
public class Repository<T> where T : class, IAuditable
{
    public void Add(T item) { _items.Add(item); }
}

var repo = new Repository<Order>();   // ← 编译期：T 被换成 Order
repo.Add(new Order { Id = "A-100", Amount = 88.8m });

// 编译产物里根本不存在 "T"，只有 "Order"
// 它不是 (Order)item 那种运行期类型转换
锚句：Repository<T> → Repository<Order> 是编译期类型实参替换，不是运行期 cast。
52. 变性默认是不变 (invariance) — string : object ≠ IBox<string> : IBox<object> 
// 直觉：string 是 object 的子类型
object o = "hello";       // ✅ OK

// 但是泛型默认不变
IBox<string> sBox = ...;
IBox<object> oBox = sBox;   // ❌ 编译错（默认 invariance）

// 想让它行，要显式 opt-in：
public interface IBox<out T> { T Read(); }   // ← 协变
IBox<object> oBox = sBox;                   // ✅ OK
锚句："string : object 是继承关系，不会自动延伸到类型实参替换关系。要开通这个延伸，必须显式标 out 或 in。"
53. out T / in T — 表层意义先记住，深层之后再啃 
public interface IProducer<out T>   // out = T 只能"出去"（作返回值）
{
    T Produce();          // ✅
    // void Take(T x);    // ❌ 不能作参数
}

public interface IConsumer<in T>    // in = T 只能"进来"（作参数）
{
    void Consume(T value);    // ✅
    // T Produce();           // ❌ 不能作返回值
}

// 协变：IProducer<string> 可当 IProducer<object> 用
// 逆变：IConsumer<object> 可当 IConsumer<string> 用
锚句：变性深层 reasoning 很多人都卡过——park 它不丢人。下次重启：IEnumerable<T> / Action<T> / Func<TIn, TOut> 三个真实接口当切入点，从"读 vs 写"直觉先入手。
54. 空 List ≠ null — 用 Count == 0 判空，别用 == null 
var empty = new List<int>();   // 有对象（~40 字节），只是没元素

if (empty == null) { /* 永远不会进 */ }    // ❌ 误区
if (empty.Count == 0) { /* ✅ 这才是判空 */ }

// 真要表达"没盒子"，让变量本身是 null：
List<int>? maybe = null;       // 需要 NRT 注解 ?
if (maybe is null) { /* ✅ */ }
锚句："有空房间" 和 "没房间" 是两件事。new List() 给你的是有空房间。
55. ?. 空条件运算符短路 — 链中任一段 null，整条返回 null 
string? name = user?.Profile?.Name;
//             ↑     ↑        ↑
//      user null → 整条 null（Profile / Name 都不会调）
//      Profile null → 整条 null
//      都非 null → 拿到真实 Name

// 别和 ?? 混用搞错：
// ?.  → null-conditional （操作修饰）
// ??  → null-coalescing  （兜底默认）
string display = user?.Name ?? "Anonymous";
锚句："?. 短路 整条断，谁先 null 谁触发。""?? 兜底 左边 null 用右边。"
56. List<List<T>> ≠ T[,] — 锯齿 vs 矩形 
// 锯齿数组 (jagged) —— 每行长度可不一样
List<List<int>> jagged = new()
{
    new() { 1, 2, 3 },
    new() { 4, 5 },        // ← 短一格也行
};
jagged[0][2]   // = 3
jagged[1][2]   // ❌ IndexOutOfRange（这一行只有 2 个）

// 矩形数组 (rectangular) —— 强制每行同长
int[,] rect = new int[2, 3] { { 1,2,3 }, { 4,5,6 } };
rect[0, 2]    // = 3   注意是逗号 [r, c]，不是 [r][c]
锚句：不规则数据（订单 / 评论 / 嵌套 JSON）→ 锯齿；固定矩形（棋盘 / 图像 / 矩阵）→ T[,]。
57. 嵌套下标越界 — 写成 m[Count][Count] 
var m = new List<List<int>> { new() { 1, 2, 3 } };
// m.Count == 1, m[0].Count == 3

// ❌ 直觉写法
var last = m[m.Count][m[0].Count];   // m[1] 就炸了 —— 合法下标只到 Count-1

// ✅ 帽子运算符
var lastSafe = m[^1][^1];            // 永远是最后一个
// ✅ 或者
var lastSafe2 = m[m.Count - 1][m[^1].Count - 1];
锚句："长度 N 的集合，合法下标只到 N-1。" 想要最后一个：[^1]。
58. dict["missingKey"] 抛 KeyNotFoundException 
var dict = new Dictionary<string, int> { ["a"] = 1 };

// ❌ 暴力取
var x = dict["nope"];   // throws KeyNotFoundException

// ✅ TryGetValue（标准姿势）
if (dict.TryGetValue("nope", out var v))
    Console.WriteLine(v);
else
    Console.WriteLine("not found");

// ✅ GetValueOrDefault（兜默认值，C# 7+）
var safe = dict.GetValueOrDefault("nope", -1);   // = -1
锚句："Dictionary 的 [] 不是数组下标——它假设你确信 key 存在。"
59. Dictionary<string, ...> 默认大小写敏感 
var d = new Dictionary<string, int> { ["alice"] = 1 };
d["Alice"];   // ❌ KeyNotFoundException —— 大小写不匹配

// ✅ 忽略大小写：初始化时传 StringComparer
var ci = new Dictionary<string, int>(StringComparer.OrdinalIgnoreCase);
ci["alice"] = 1;
ci["Alice"];  // ✅ = 1

// 常用 comparer：
// OrdinalIgnoreCase           ← 推荐：性能最好、文化无关
// CurrentCultureIgnoreCase    ← 跟着系统区域；UI 字符串才用
// InvariantCultureIgnoreCase  ← 跨平台稳定的文化无关比较
锚句："string key 默认 case-sensitive。初始化那一刻不传 comparer，后面就锁死了。"
60. decimal d = 1000.0; 编译错 — 字面量必须带 m 
// ❌ 1000.0 是 double 字面量，不能隐式转 decimal
decimal price = 1000.0;          // CS0664

// ✅ m 后缀 = decimal 字面量
decimal price = 1000.0m;
decimal big   = 1_000_000.50m;   // 下划线只是给人看的，编译器忽略

// 数字字面量后缀全家桶：
//   m / M  → decimal （money）
//   f / F  → float
//   d / D  → double（默认，可省）
//   L      → long
//   u / U  → uint
//   ul UL  → ulong

// 💰 金额永远 decimal，never double：
double  a = 0.1 + 0.2;     // = 0.30000000000000004   ❌
decimal b = 0.1m + 0.2m;   // = 0.3                   ✅
锚句："m = money。钱用 decimal，不用 double，否则 0.1 + 0.2 就给你来个 0.30000000000000004。"
61. 读 Func<...> 时眼睛跑去看"中间最显眼的东西" — 忘了找返回类型 
Func<Dictionary<string, List<int>>, List<string>> findHighScorers;
//   ──────────── 输入 ────────────  ── 返回 ──
//                                    ↑
//                                    第一眼应该看这里！

// ❌ 易错心理路径：
//    "Dictionary、string、List ……" 一顿目光扫，结果说返回类型是 List
//    （那其实只是 Dictionary 的 value 类型）

// ✅ 正确目光顺序：
//    1. 数最外层 < 和 > 找配对
//    2. 直接跳到最后一个顶层 , 后面那块
//    3. 那就是返回类型
//    4. 前面所有顶层逗号分隔的块 = 输入参数
锚句："读 Func<...>，目光直接弹到最后一个泛型参数。前面全是噪音。"（口诀 #43）
62. 嵌套泛型逗号有两种身份 — 数 < > 配对深度 
Func<Dictionary<string, List<int>>, List<string>>
//   ^         ^      ^      ^   ^^               ^
//   d=1       d=2    d=2    d=1 d=0              d=0
//                    ↑                ↑
//                    Dictionary 内部   Func 的真正分界
//                    的逗号            （d=0 时的逗号）

// ✅ 解析规则：
//   - 每遇到一个 <，depth += 1
//   - 每遇到一个 >，depth -= 1
//   - 只有 depth == 1（顶层 Func<...> 内）时遇到的逗号
//     才属于 Func 的参数分隔符
锚句："嵌套泛型的逗号，归谁要看 <> 配对的深度。" 这一条把"剥洋葱解码法"变成可机械执行的步骤。
63. 多次枚举 IEnumerable<T> — 每用一次配方就重跑一次 
var query = users
    .Where(u => SlowDbLookup(u.Id))         // 配方，没跑
    .Select(u => u.Name);

if (query.Any())                             // ❌ 跑了一次 (N 次 DB)
{
    Console.WriteLine(query.Count());        // ❌ 又跑了一次 (N 次 DB)
    foreach (var n in query) ...             // ❌ 第三次 (N 次 DB)
}

// ✅ 加一行物化，3N 变 N
var names = users.Where(...).Select(...).ToList();
if (names.Count > 0)
{
    Console.WriteLine(names.Count);
    foreach (var n in names) ...
}
锚句："IEnumerable<T> 是配方不是菜。要复用先 .ToList() 煮熟。"（口诀 #45）
ReSharper / Rider 警告：Possible multiple enumeration of IEnumerable
64. 把 Where()/Select() 的结果当 List<T> 
List<int> nums = [1, 2, 3];
var filtered = nums.Where(n => n > 1);

filtered.Add(99);     // ❌ IEnumerable<int> 没有 Add
filtered[0];          // ❌ 没有索引器
filtered.Count;       // ❌ 没有 Count 属性 (Count() 方法可以，但每次重跑)

// ✅ 想用 List API 必须显式物化
List<int> list = nums.Where(n => n > 1).ToList();
list.Add(99);  list[0];  var c = list.Count;
锚句："LINQ 操作返回 IEnumerable<T>，不是你输入的具体类型。能 foreach ≠ 是 List。"（口诀 #44）
65. var + 集合表达式 [...] 或目标类型 new(...) — 编译失败 
// ❌ 右边没类型可推断
var a = [1, 2, 3];                  // CS9176 / CS8754
var b = new(1, "Alice");            // CS8754

// ✅ 左边写明类型 → 编译器目标推断
List<int> a = [1, 2, 3];            // 集合表达式
User b = new(1, "Alice");           // target-typed new

// ✅ 或显式写完类型
var a = new List<int> { 1, 2, 3 };
var b = new User(1, "Alice");
锚句："[...] 和 new(...) 都要左边告诉它要什么形状，不能光靠右边裸跑。"（口诀 #48）
66. 假设 GroupBy 袋子顺序固定 — 接口不承诺 
// LINQ to Objects 实际给的是"首次出现"序，但仅是实现细节
var groups = orders.GroupBy(o => o.Status);
foreach (var g in groups)
    Console.WriteLine(g.Key);   // 当前实现：Paid, Cancelled, Pending

// ❌ 同样代码在 EF Core / PLINQ / LINQ to XML 可能顺序不同
//    甚至 LINQ to Objects 升级也可能变

// ✅ 要顺序就显式 OrderBy
var stable = orders
    .GroupBy(o => o.Status)
    .OrderBy(g => g.Key);       // 字典序：Cancelled, Paid, Pending
锚句："GroupBy 的袋子顺序，接口不承诺。要稳就显式 OrderBy。"（口诀 #50）
67. 同一 assembly 顶级语句 + Main() 共存 — 静默选顶级语句 
// 文件 1: Program.cs (dotnet new console 默认)
Console.WriteLine("Hello, World!");      // ← 顶级语句

// 文件 2: MyDemo.cs (你自己写的 demo)
public static class Program
{
    public static void Main()             // ❌ 编译器 warning CS7022:
    {                                     //    The entry point of the program
        Console.WriteLine("Demo!");       //    is global code; ignoring 'Main()'
    }
}

// dotnet run → 输出 "Hello, World!"，不是 "Demo!"
//   exit 0，只有 warning，没有 error，最容易漏掉

// ✅ 解决方案 (4 选 1)
// A. 删掉默认 Program.cs，让 demo 的 Main 成为唯一入口
// B. 直接编辑 Program.cs，不要新开文件
// C. 把 demo 文件重命名成 Program.cs
// D. 在 .csproj 加 StartupObject，强制硬错而不是 warning：
//    <PropertyGroup>
//      <StartupObject>MyNamespace.Program</StartupObject>
//    </PropertyGroup>
锚句："一个 assembly 一个入口点。顶级语句存在时编译器先选它，Main() 被降级成普通静态方法，只有 warning CS7022，不报错 —— 静默踩坑。"（口诀 #52）
