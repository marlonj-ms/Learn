1. 一句话总结（10 条核心要点）

  1. 扩展方法三铁律：static class + static method + 第一参数加 this
  2. this 是语法糖：n.IsEven() 编译后 ≡ MyExtensions.IsEven(n)
  3. 查找规则：编译器靠 using 命名空间 + this 参数类型定位，跟 class 名字无关
  4. 重名冲突：编译器报错（ambiguous），不会自动选
  5. 能扩展一切：基础类型 / 引用类型 / 接口 / 泛型 / 委托 / object
  6. Lambda 是个值：和 5、"hello" 地位相同，可赋值/传参/return
  7. 方法可返回 lambda（"函数工厂"模式）
  8. Lambda 形参 = 方法形参，调用时才拿到具体值
  9. 闭包：lambda 能"记住"外层方法的变量
  10. 函数组合：a.AndThen(b) = 先 a，然后 b（内层括号 = 先执行）

2. 扩展方法 — 三铁律

public static class MyExtensions     // ① 必须 static class
{
    public static bool IsEven(this int n)   // ② static method
        => n % 2 == 0;                      // ③ this 第一参数
}
规则说明
① 类必须 static class普通 class 即使方法是 static 也不行
② 方法必须 static因为它本质就是一个静态调用
③ 第一参数加 thisthis 那个参数的类型 = 被扩展的类型

3. this 是语法糖（最关键认知）

今天最大的卡点之一：「这个看起来就是定义了一个 static class 和 method，默认就可以 MyExtensions.IsEven() 调用，为啥 n.IsEven() 可以 work？」

答案：this 关键字告诉编译器：「允许我用点语法调用」。编译器在底层把 n.IsEven() 翻译成 MyExtensions.IsEven(n)。两种写法编译出来完全一样。

// 版本 A：带 this
public static bool IsEven(this int n) => n % 2 == 0;

// 版本 B：不带 this
public static bool IsOdd(int n) => n % 2 != 0;

int x = 5;

// 版本 A 两种写法都行：
x.IsEven();                  // ✅ 编译器认识
MyExtensions.IsEven(x);      // ✅ 也行

// 版本 B 只能这样：
MyExtensions.IsOdd(x);       // ✅ 行
x.IsOdd();                   // ❌ 编译错误！没有 this 就不能点语法
为啥要发明这个语法糖？ 为了链式调用好看。
list.Where(...).Select(...) 比 Enumerable.Select(Enumerable.Where(list, ...), ...) 优雅一万倍。

4. 编译器如何"找到"扩展方法

卡点：「public static class MyExtensions 这个和数字毫无关联，为什么 x.IsEven() 知道找这个 class？」

答案：编译器不看 class 名字。它做的事是：

看到 x.IsEven()
  ↓
1. 先看 int 类型自己有没有 IsEven 方法？→ 没有
  ↓
2. 扫描当前文件所有 using 引入的命名空间
  ↓
3. 在这些命名空间里找：所有 static class 中，
   带 this 的、第一个参数类型匹配 int（或 int 的父类）的方法
  ↓
4. 找到了 MyExtensions.IsEven(this int n) ✅
  ↓
5. 把 x.IsEven() 翻译成 MyExtensions.IsEven(x)
这就是为什么：不写 using System.Linq; 就用不了 list.Where(...)。 Where 定义在 System.Linq.Enumerable 静态类里， 没引入命名空间，编译器扫描不到。

5. 重名冲突 — 编译器不会猜

public static class A { public static bool IsEven(this int n) => n % 2 == 0; }
public static class B { public static bool IsEven(this int n) => n % 3 == 0; }

using A;        // 只引入 A → 调 A 的（n % 2）
using B;        // 只引入 B → 调 B 的（n % 3）

// 两个都引入：
using A;
using B;
x.IsEven();     // ❌ 编译错误：ambiguous call

// 必须显式：
A.IsEven(x);    // ✅
B.IsEven(x);    // ✅
原则：编译器从不猜测。遇到不明确就报错，让你显式指定。

6. 多参数扩展方法

public static int AddBy(this int age, int y) => age + y;

int m = 10;
m.AddBy(3);     // → 13

编译器翻译：

MyExtensions.AddBy(m, 3)
//                 ↑  ↑
//              this 普通参数
参数位置怎么传
this 那个（第一个）由"点前面的对象"自动传入
后面的所有参数调用时在括号里显式传入
这就是 LINQ 的工作方式： 
public static IEnumerable<T> Where<T>(
    this IEnumerable<T> source,    // ← 自动从"点前"对象拿
    Func<T, bool> predicate)        // ← 括号里显式传

list.Where(n => n > 5)
//   ↑     ↑
// source predicate

7. 扩展任意类型（包括委托）

卡点：「如果 public static string IsEven(this Func<int, bool>) 也可以扩展一个 Func 的扩展？」

答案：完全可以！扩展方法的能力远超你想象：

// ① 基础类型
public static int Squared(this int n) => n * n;

// ② string
public static bool IsEmail(this string s) => s.Contains("@");

// ③ 集合
public static T SecondOrDefault<T>(this IEnumerable<T> source)
    => source.Skip(1).FirstOrDefault();

// ④ 委托（最酷的）
public static string Describe(this Func<int, bool> predicate)
{
    bool resultOn5 = predicate(5);
    return $"对 5 的判断结果是：{resultOn5}";
}

Func<int, bool> isPositive = n => n > 0;
isPositive.Describe();   // → "对 5 的判断结果是：True"

// ⑤ object（所有类型）
public static string ToJson(this object obj) => /* ... */;
注意：Func 泛型参数只写类型，不带参数名。
Func<int, bool> ✅    Func<int x, bool y> ❌

8. = vs => 三种身份

卡点：「我有的时候看到 n => second(first(n));，有的时候看到 => n => second(first(n));，有的时候看到 = n => n * 10;。我很困惑。」
符号位置含义例子
= 变量声明后 赋值运算符 int x = 5;
=> 参数列表后 n => Lambda 箭头 n => n + 1
=> 方法/属性签名后 ) => 表达式体（语法糖） int F(int n) => n + 1;

经典：一行里出现两个 =>

public Func<int, int> MakeAdder(int n) => x => x + n;
//                                     ↑   ↑
//                          表达式体（return）  Lambda 箭头

这不是错误！它等价于：

public Func<int, int> MakeAdder(int n)
{
    return x => x + n;
}

识别口诀 ⭐

  • 左边是参数 → Lambda 箭头
  • 左边是方法/属性签名 → 表达式体语法糖（替代 { return ... }）

实战识别（今天答对了 5/5）

// 例 1
Func<int, int> a = x => x + 1;            // → Lambda 箭头

// 例 2
public int Add(int x, int y) => x + y;     // → 表达式体

// 例 3
public Func<int, int> MakeAdder(int n) => x => x + n;
//                                       ↑   ↑
//                                   表达式体  Lambda 箭头

// 例 4
list.Where(x => x > 5);                    // → Lambda 箭头

// 例 5
public string Name => "Tom";               // → 属性的表达式体！

9. 表达式体的陷阱

表达式体 => 后面只能跟一个表达式，不能跟代码块 { ... }！ 
// ✅ 一行表达式
public int Square(int n) => n * n;

// ❌ 不能有大括号
public int Square(int n) =>
{
    int result = n * n;
    return result;
};   // 编译错误！

// ✅ 多行必须用大括号 + return
public int Square(int n)
{
    int result = n * n;
    return result;
}

10. Lambda 是个值（心智模型升级）⭐

今天最大的认知突破。

普通的值你都见过：

int x = 5;            // 5 是一个 int 值
string s = "hello";   // "hello" 是一个 string 值
Func<int, int> f = n => n + 5;
// ────────────────  ──────────────
//   左：变量          右：lambda（也是一个值！）

Lambda 就是一个"没名字的方法"。它和普通方法是同一种东西：

// 写法 A：完整方法（有名字）
int Add5(int n)
{
    return n + 5;
}

// 写法 B：lambda（没名字）—— 等价！
n => n + 5
普通方法Lambda
有名字没名字
必须先定义后调用可以"现造现用"
参数列表 (int n)参数列表 n =>
函数体 { return ...; }函数体 ...（在 => 后）

既然是值，它就能：

Func<int, int> f = n => n * 2;       // ✅ 赋值给变量
list.Where(n => n > 5);              // ✅ 作为参数
return n => n + 5;                   // ✅ 从方法 return（下一节）

11. 方法可以 return lambda（函数工厂）

public static Func<int, int> AddBy(int amount)
{
    return n => n + amount;
}

Func<int, int> add5 = AddBy(5);     // 工厂生产"加 5"的函数
Func<int, int> add7 = AddBy(7);     // 工厂生产"加 7"的函数

add5(100);   // → 105
add7(100);   // → 107

等价的"笨"写法（自己拆解出来的）

今天的精彩瞬间：「其实可以方法拆解为下面这样⬇️」 
public static Func<int, int> AddBy(int amount)
{
    Func<int, int> x = n => n + amount;   // ← 先创建 lambda，存到变量 x
    return x;                              // ← 再返回 x
}

能这样拆，说明你看穿了：return n => n + amount; 不是神奇语法，就是"return 一个值"，只不过这个值是 lambda。

就像 return 5;（返回 int）、return "hi";（返回 string），
return n => n + 5;（返回 Func<int,int>）—— 三者地位完全平等。

12. Lambda 形参 = 等被调用时才拿值

卡点：「return n => n + amount; 这个 n 哪里来的？」

这个 n 不是任何已存在变量。它是 lambda 自己声明的形参，就像普通方法的形参：

int Foo(int n) { return n + 1; }   // 这个 n 也不是已存在变量
                                    // 它是签名里的形参，等调用者传值
n => n + 1                          // lambda 的 n 完全一样

时间线（用 AddBy(5) → add5(100) 举例）

时间事件n 的状态
1AddBy(5) 进入n 不存在
2return n => n + amount;n 是占位符（参数声明）
3add5 = ... 赋值n 还是占位符
4add5(100) 调用n 终于 = 100
5执行 n + amount100 + 5 = 105
钉到脑子里：Lambda 是"还没被调用的函数"。参数 n 在调用之前是占位符。

13. 闭包初探（捕获外层变量）

看 AddBy 的奇怪之处：

public static Func<int, int> AddBy(int amount)
{
    return n => n + amount;   // ← 用到了 amount
}

Func<int, int> add5 = AddBy(5);
// AddBy 调用结束了，按理说局部变量 amount 应该消失
// 但 add5(100) 居然还能算出 105 —— amount 被"记住"了！
这就是闭包（closure）：Lambda "捕获"了它外层作用域的变量，即使外层方法已返回，lambda 还能访问这些变量。

闭包是函数式编程的核心，今天先记住名字，将来专门讲。

14. AndThen — 函数组合

public static Func<int, int> AndThen(
    this Func<int, int> first,
    Func<int, int> second)
{
    return n => second(first(n));
}

Func<int, int> addOne  = x => x + 1;
Func<int, int> times10 = x => x * 10;

addOne.AndThen(times10)(5);   // → (5+1)*10  = 60
times10.AndThen(addOne)(5);   // → (5*10)+1  = 51   注意顺序不同！

慢动作回放：addOne.AndThen(times10)

Step 1: addOne.AndThen(times10) 进入 AndThen
        first  = addOne
        second = times10

Step 2: 创建 lambda：n => times10(addOne(n))
        （first、second 被闭包捕获）

Step 3: 返回 lambda 给 combined

Step 4: combined(5)
        → times10(addOne(5))
        → times10(6)
        → 60

链式 pipeline

Func<int, int> pipeline = addOne
    .AndThen(times10)
    .AndThen(x => x - 5);

pipeline(5);   // ((5+1)*10) - 5 = 55

15. 为什么 second(first(n)) 不能改成 first(second(n))

今天的精彩调试：曾经想把方法体改成 return n => first(second(n));，被否决了。

原因：方法名是 AndThen（然后），语义是"先 a，然后 b"。必须让名字和执行顺序一致。

return n => second(first(n));
//                 ────────
//                 最内层 = 最先执行 = "first"
记住：C# 求值顺序 = 从内层括号往外。
所以"想让谁先执行"，就把它放在最内层。
名字叫 first 的那个，必须放在最内层，否则名字就反了。

16. 扩展方法 = 自动传 + 显式传

今天的关键顿悟：「AndThen 里面其实是有 2 个参数的，但实际 call 时（）里面只有一个。first 通过 s.AndThen(t) 的 s 传递进去。t 是 second。」

完全正确！这就是扩展方法的核心机制：

public static Func<int, int> AndThen(
    this Func<int, int> first,    // ← 第 1 个参数（带 this）
    Func<int, int> second)         // ← 第 2 个参数（普通）

// 调用：
s.AndThen(t)
// │       │
// │       └── 显式传 → 接收方是 second
// └── 自动塞给 this 参数 → 接收方是 first

// 编译器实际生成：
ExtensionsClass.AndThen(s, t);
//                      ↑  ↑
//                   first  second

17. 实战练习中暴露的语法盲点

陷阱 ①：返回类型对不上

// ❌ 写错的版本
public static string WordCount(this string words) => words.Length;
//                   ↑                                ↑
//                返回 string                       返回的是 int

// ✅ 正确
public static int WordCount(this string sentence) =>
    sentence.Split(' ').Length;

陷阱 ②：IEnumerable<T> 没有 .Length 也没有 .Count 属性

// ❌ 写错的版本
public static int Average2(this IEnumerable<int> y)
{
    int sum = 0;
    foreach (int x in y) sum += x;
    return sum / y.Length;   // ❌ IEnumerable 没有 Length！
}

// ✅ 正确：边遍历边计数
public static int Average2(this IEnumerable<int> nums)
{
    int sum = 0, count = 0;
    foreach (int x in nums) { sum += x; count++; }
    return sum / count;
}

长度三表（重要参考）

类型怎么拿长度类型
int[] / string.Length属性
List<T> / Dictionary<K,V> / HashSet<T>.Count属性
IEnumerable<T>.Count()LINQ 方法（遍历整个序列）
原因：IEnumerable<T> 可能是无限序列，所以没有"长度"的概念。
