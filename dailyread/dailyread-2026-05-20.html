1. AAA 模式 + xUnit 最小骨架

[Fact]
public void AboveThreshold_RaisesEvent()
{
    // Arrange — 建对象、订阅事件、准备标志变量
    var sensor = new TemperatureSensor(threshold: 100.0);
    bool eventFired = false;
    sensor.ThresholdExceeded += (s, e) => eventFired = true;

    // Act — 通常一行，触发被测路径
    sensor.RecordReading(101);

    // Assert — 自动判定 PASS/FAIL
    Assert.True(eventFired);
}
xUnit 用 反射 + metadata 发现 [Fact] 方法，并给每个测试方法新建一个测试类实例，所以测试之间永远不会共享状态。

2. 解决方案 / 项目 / 引用

TemperatureSensor.slnx                    ← 容器，不参与运行
├── TemperatureSensor.Core/               ← classlib，无 Main，纯逻辑
│   └── TemperatureSensor.cs              ← 零 Console.WriteLine
└── TemperatureSensor.Tests/              ← xunit 模板 + ProjectReference
    └── TemperatureSensorTests.cs
产品代码没有 Main，必须被 host（控制台 / Web API / xUnit runner）消费。引用是给编译器的，反射才是 xUnit 用的。

3. [Theory] + [InlineData] —— 等价类覆盖

[Theory]
[InlineData(150.0, true)]      // above 等价类
[InlineData(100.0, false)]     // boundary 边界（关键！）
[InlineData( 50.0, false)]     // below 等价类
public void RecordReading_...(double celsius, bool shouldRaise)
{
    var sensor = new TemperatureSensor(100.0);
    bool fired = false;
    sensor.ThresholdExceeded += (s, e) => fired = true;

    sensor.RecordReading(celsius);

    Assert.Equal(shouldRaise, fired);
}
边界覆盖缺口：只测 99/101 没人守门 > 还是 >=。把 100.0 这个边界值放进 [InlineData] 才能抓住运算符变化。

4. 对象 spy + payload 验证

ThresholdExceededEventArgs? captured = null;     // ← T? 声明可空
sensor.ThresholdExceeded += (s, e) =>
{
    eventFired = true;
    captured = e;                                  // ← lambda 闭包扣留 e
};

sensor.RecordReading(101);

Assert.NotNull(captured);
Assert.Equal(101, captured!.Reading);              // ! = null-forgiving
Assert.Equal(100, captured.Threshold);
captured = null 不能写 var captured = null; —— null 没类型，编译器推不出来。必须显式声明 ThresholdExceededEventArgs?。
测试盲点（test blind spot）：只检 eventFired == true 是"伪绿"。今天 cycle 3 把 new ThresholdExceededEventArgs(celsius, _threshold, ...) 反成 (_threshold, celsius, ...)，5 个测试里只有 1 个抓到，因为只有它在断言 captured.Reading。

5. Assert.Throws<T> —— 异常测试

// 产品代码：防御性输入校验（快速失败）
public void RecordReading(double celsius)
{
    if (double.IsNaN(celsius) || double.IsInfinity(celsius))
    {
        throw new ArgumentException(
            "Reading must be a finite number.",
            nameof(celsius));        // ← nameof 重构友好
    }
    if (celsius > _threshold) OnThresholdExceeded(...);
}

// 测试
[Fact]
public void RecordReading_NaN_ThrowsArgumentException()
{
    var sensor = new TemperatureSensor(100.0);
    Assert.Throws<ArgumentException>(() => sensor.RecordReading(double.NaN));
}
API含义
Assert.Throws<T>精确类型匹配，子类不算
Assert.ThrowsAny<T>接受 T 或其子类
返回值捕获到的 exception 对象，可继续 Assert.Contains(...)
异常层级：Exception → SystemException → ArgumentException → ArgumentNullException / ArgumentOutOfRangeException。
"参数不合法" 通用情形 → ArgumentException；具体 null → ArgumentNullException；超范围 → ArgumentOutOfRangeException。

6. [MemberData] —— 数据源外置

[Theory]
[MemberData(nameof(ThresholdTestData))]    // ← nameof 引用，重构友好
public void RecordReading_...(double celsius, bool shouldRaise) { ... }

public static IEnumerable<object[]> ThresholdTestData =>
    new List<object[]>
    {
        new object[] { 200.0,  true  },
        new object[] { 100.001, true },     // 浮点边界微高
        new object[] { 100.0,  false },     // 边界
        new object[] { 99.999, false },     // 浮点边界微低
        new object[] { -40.0,  false },     // 负温度
    };

7. => 的双重身份 ⚠️

位置例子身份
值的位置Func<int,int> f = x => x*2;Lambda
成员声明位置public int X => 42;表达式体成员（expression-bodied member）
表达式体成员 = { return ...; } 的简写，C# 6 引入。可用在属性 / 方法 / 索引器 / 构造函数 / 析构函数。

8. [MemberData] 三种成员形式

// 形式 1：属性 + 表达式体（最常用 ⭐⭐⭐⭐⭐）
public static IEnumerable<object[]> Data => new[] { ... };
//                                       ↑↑ 每次访问 Data 都执行右边

// 形式 2：字段 + 赋值（很少用）
public static IEnumerable<object[]> Data = new[] { ... };
//                                       ↑ 类加载时执行一次

// 形式 3：方法 + 表达式体
public static IEnumerable<object[]> Data() => new[] { ... };
//                                        ↑↑↑ 多了一对括号 = 方法
判断口诀：①有 () → 方法；②无 () 看符号：=> → 属性，= → 字段。

9. xUnit1024 —— 测试方法不能重载

// ❌ 编译错误：xUnit1024
[Theory] void Add_Works(int a, int b) { ... }
[Theory] void Add_Works(double a, double b) { ... }

// ✅ 改名
[Theory] void Add_Works_Int(int a, int b) { ... }
[Theory] void Add_Works_Double(double a, double b) { ... }
这不是 C# 编译器报的错，是 xUnit 的 Roslyn 分析器。原因：CI 用方法名作测试唯一 ID，重名会撞车。产品代码可以重载，测试不行。

10. fail-fast + 调试心智模型

[Fact] Test_Method:
    Assert 1   ✅ PASS
    Assert 2   ✅ PASS
    Assert 3   ❌ FAIL  ← 抛 AssertException，整个方法在这里停
    Assert 4   ⚫ 未执行
    Assert 5   ⚫ 未执行
测试只报症状（WHAT 值不对），不报病因（WHERE 哪行 bug）。Stack trace 指向 Assert 行，不指向产品代码 —— 例外：产品代码抛异常时，stack trace 会直接刺穿到产品代码。

11. 反复踩的坑（今天新增）

  1. var x = null; ❌ 编译不过 —— null 没类型，必须显式 ThresholdExceededEventArgs? x = null;
  2. 测试方法不能重载（xUnit1024）—— CI 用方法名作 ID，重名撞车。
  3. Assert.Throws<T> 是精确类型匹配，子类不算。要接受子类用 Assert.ThrowsAny<T>。
  4. x == double.NaN 永远 false（IEEE 754：NaN != NaN）。要用 double.IsNaN(x)。
  5. => 在 Lambda 和表达式体里是两个东西。看位置区分。
  6. Assert.Equal(expected, actual) 参数顺序铁律 —— 反了报错信息会误导。
  7. 本地变量 / 参数命名必须 camelCase（celsius），不能 sensortemp。
  8. new object[] { ... } 强制成 object[]；new[] { 1, 2 } 会推断成 int[]。[MemberData] 必须前者。
  9. [MemberData] 引用的成员必须 public static，私有或实例都不行。
  10. 多目标 csproj 三件事都得改对：标签必须 <TargetFrameworks>（复数 s），分隔符必须分号 ;（不是逗号），DLL 落到 lib/<TFM>/ 里。错一个就 NETSDK1046。
  11. dotnet pack 默认元数据是垃圾：<authors> 自动取项目名，<description> 是字面字符串 "Package Description"。这是 feature 不是 bug —— 强迫你注意没填。
  12. <Version> + <PackageReleaseNotes> + 注释是三方契约：版本号从 1.1.0 跳到 1.1.1 是 PATCH（纯修 bug），不再是 MINOR（加功能）。三处必须同步改。

SDK三义：①.NET SDK（toolchain）②服务 SDK（Azure/AWS 一族包）③厂商库你用的 dotnet CLI 来自 .NET SDK
Library vs Framework 看控制流：
库 = 你叫它（YourCode → Library）
框架 = 它叫你（Framework → YourCode，即"控制反转"）
xUnit 是框架，因为是它通过反射调用你的 [Fact] 方法。你自己的 TemperatureSensor.Core.dll 是库。

SemVer 三段位

段位变化承诺
1.0.0 → 1.0.1（PATCH）纯 bug 修复什么都没加，只修
1.0.0 → 1.1.0（MINOR）加了新 API，旧 API 不变你不用动代码也能升
1.0.0 → 2.0.0（MAJOR）删 / 改名 / 改签名公开 API消费者可能要改代码

13. 晚上加场 · dotnet pack 三轮实战

# 在 TemperatureSensor.Core 目录下
dotnet pack -c Release
轮次csproj 状态产物大小
Pass 1裸跑，无元数据TemperatureSensor.Core.1.0.0.nupkg（默认假装 1.0.0）4,303 B
Pass 2加 PackageId/Version/Authors/Desc/License/Tags/NotesLearning.TemperatureSensor.Core.1.1.0.nupkg5,073 B

多目标三大坑（NETSDK1046）： 
❌ <TargetFramework>net10.0,net8.0</TargetFramework>     // 逗号、单数
❌ <TargetFramework>net10.0;net8.0</TargetFramework>     // 分号对了，但还是单数
✅ <TargetFrameworks>net10.0;net8.0</TargetFrameworks>   // 复数 s + 分号
报错信息会直接告诉你："use the 'TargetFrameworks' property instead" —— 读 SDK 报错信息，往往就是答案。

14. 晚上加场 · IL 预览（已 defer）

今天用 ilspycmd 看了一眼 IL，主动 defer 到以后。学语言不必先懂 IL，但下面这两条"实质"值得记。

自动属性 = 一个隐藏字段 + 一个方法

public double Reading { get; }   // C# 里看到的

// IL 里实际是：
private readonly double <Reading>k__BackingField;   // 字段（<> 在 C# 里非法，编译器专用）
public double get_Reading() {                        // 方法（3 条 IL：ldarg.0; ldfld; ret）
    return <Reading>k__BackingField;
}

事件 = 一个委托字段 + 两个方法（含 CAS 锁循环）

public event EventHandler<T>? Foo;   // C# 里看到的（一行）

// IL 里实际是：
private EventHandler<T> Foo;                     // 委托链字段
public void add_Foo(EventHandler<T> value) {     // += 调用的方法
    // ~30 行 CAS 循环：Interlocked.CompareExchange + Delegate.Combine
    // 保证多线程并发订阅时不丢 handler
}
public void remove_Foo(EventHandler<T> value) {  // -= 调用的方法
    // 同样的 CAS 循环，但用 Delegate.Remove
}
一个 public event Foo; 一行代码，编译器给你生成 ~30 行线程安全 CAS 循环。这就是为什么 C# 让你写 += 而不是手动维护委托链。

C# 表象 → IL 实质（速查）

C# 写法IL 实质
public T X { get; }backing field + get_X() + .property metadata
public event T Foo;backing field + add_Foo(CAS) + remove_Foo(CAS) + .event metadata
lambda编译器生成的匿名类 + Invoke 方法（闭包变量是该类的字段）
async/await状态机类 + MoveNext()（之后讲）
using x = ...;try { ... } finally { x.Dispose(); }
foreach (var x in xs)GetEnumerator() / MoveNext() / Current + try/finally
?.null 检查 + 分支
nameof(x)编译期字符串字面量，运行时零成本
