DI + ILogger 实操 + Lambda 三层拆解

2026-05-21 · 5 分钟复习 · 配套：完整 Summary（22 题） · 晨间理论 · Program.cs · Service 代码

今天主线：晨间把 DI + ILogger + 三种生命周期建好心智模型 → 午后写出第一个能跑的 DI 项目（ServiceCollection + AddLogging + AddTransient + GetRequiredService）→ 下午把卡了半天的 services.AddLogging(builder => {...}) 这一行从“烧脑”拆到“透明”：lambda、泛型委托、扩展方法三层全打通。
结果：第一次跑通 DI 容器，看到了 info: / warn: 真实日志输出。

1. 业务数据 vs 依赖项

角色放哪里例子
业务数据（business data）方法参数Order order、double celsius
依赖项（dependency）构造函数参数 → 私有只读字段ILogger<T>、IOrderRepository
DI 不是“类不声明依赖”，而是 类清晰声明依赖，但不创建具体依赖。构造函数说“我需要什么”，容器说“我给你”。

2. 最小 DI host（6 步）

using DI_Logging_MinHost;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Logging;

// 1. 建注册清单
var services = new ServiceCollection();

// 2. 注册日志基础设施（console 输出）
services.AddLogging(builder =>
{
    builder.AddConsole();
    builder.SetMinimumLevel(LogLevel.Information);
});

// 3. 注册业务服务
services.AddTransient<TemperatureSensorService>();

// 4. 构建容器（registration → container）
using var provider = services.BuildServiceProvider();

// 5. 从容器解析对象（这一步才真正创建对象）
var sensor = provider.GetRequiredService<TemperatureSensorService>();

// 6. 调用业务方法
sensor.Record(72.5);
sensor.Record(101.3);
AddTransient<T>() 不会立刻创建 T，只是注册“以后能 new、用什么 lifetime”。真正创建发生在 GetRequiredService<T>()，容器读取构造函数自动注入依赖。

3. ILogger<T> 与结构化日志

public sealed class TemperatureSensorService
{
    private readonly ILogger<TemperatureSensorService> _logger;

    public TemperatureSensorService(ILogger<TemperatureSensorService> logger)
    {
        _logger = logger;             // 构造函数注入
    }

    public void Record(double reading)
    {
        // ✅ 结构化日志（structured logging）：保留字段名 + 字段值
        _logger.LogInformation("The temperature sensor reported {Reading}", reading);

        if (reading > 100)
            _logger.LogWarning("Temperature is too high: {Reading}", reading);
    }
}
易踩坑：
1. $"...{reading}" 是字符串插值 → 日志里只剩文本，结构化字段丢失。
2. "...{Reading}", reading 才是结构化日志 → 保留 Reading 字段名和值。
3. ILogger<T> 的 T 是日志类别（log category），不是消息类型！

结构化日志按位置匹配，不按变量名

_logger.LogInformation("User {UserId} ordered {OrderId}", orderId, userId);
// {UserId}  → orderId  (按位置)
// {OrderId} → userId   (按位置)
// 字段错位 → 写反顺序会日志串数据

4. 三种 Service Lifetime 决策树

Lifetime创建时机典型用法
Transient每次 resolve 都新建轻量无状态计算器
Scoped同 scope 内一个实例（Web API 默认 = per HTTP request）CurrentUserContext、DbContext
Singleton整个容器一个实例（应用关闭时 dispose）SemaphoreSlim(10) 全局限流器、配置缓存
两个真实陷阱：
1. CurrentUserContext 注册成 Singleton → Alice 的 UserId 会泄漏给 Bob，严重安全风险。
2. SemaphoreSlim(10) 注册成 Scoped → 每个 request 自己 10 个并发，全局限制失效。
Scoped ≠ per user。默认 scope = per HTTP request。用户身份跨请求由 cookie / JWT / session 保存，不是靠 Scoped 对象长期活着。

5. Lambda 三件套 ⭐ 今天的最大收获

Action<int> log = n => Console.WriteLine(n);
//     <>              ↑参数  ↑方法体
log(42);
// () 委托调用
语法本质
Action无参、无返回值的委托类型
Action<T>接受一个 T、无返回值的委托类型（泛型）
lambda 参数 n由 lambda 自身定义；类型从委托签名推断
没有 return因为 Action<T> 返回 void，不是错误
没有 {}表达式 lambda（expression lambda），只有单一表达式

烧脑点 1：builder 是哪儿定义的？

services.AddLogging(builder =>
{
    builder.AddConsole();
});
builder 不是“缺乏定义”，它是 lambda 自己的参数。类型从 AddLogging 的签名 Action<ILoggingBuilder> 推断为 ILoggingBuilder。等价于写命名方法： 
void Configure(ILoggingBuilder builder) { builder.AddConsole(); }
services.AddLogging(Configure);

烧脑点 2：lambda 是谁在调用？

// AddLogging 内部大致这样（伪代码）：
public static IServiceCollection AddLogging(
    this IServiceCollection services,
    Action<ILoggingBuilder> configure)
{
    var b = new LoggingBuilder(services);
    configure(b);     // ← 你的 lambda 这一刻才执行
    return services;
}
这叫配置回调（configuration callback）：框架内部 new 出 builder，然后调用你的 lambda 让你配置它。

6. <> vs () 终极区分 ⚠️

符号用途例子
<>泛型类型实参 —— 给“类型”加类型参数List<int>、Action<int>、ILogger<OrderService>
()调用括号 —— 给“方法/委托”传值参数WriteLine(x)、log(42)、AddLogging(b => ...)
Action<int> 是类型，不是方法签名。所以是 <int>，不能写成 Action(int x)。

7. 扩展方法（extension method）—— this 在哪？

public static class StringExtensions
{
    public static int LengthSquared(this string s) => s.Length * s.Length;
    //                              ↑↑↑↑ 关键：把这个方法变成扩展方法
}

"hello".LengthSquared();
// 编译期改写为：StringExtensions.LengthSquared("hello");
this 写在定义里的第一个参数前，调用时不写。services.AddLogging(...) 不写 this，是因为 AddLogging 的定义里已经写了 this IServiceCollection services。

8. 跑起来的真实输出

info: DI_Logging_MinHost.TemperatureSensorService[0]
      The temperature sensor reported 72.5
info: DI_Logging_MinHost.TemperatureSensorService[0]
      The temperature sensor reported 101.3
warn: DI_Logging_MinHost.TemperatureSensorService[0]
      Temperature is too high: 101.3
段含义
info / warn日志等级（log level）
DI_Logging_MinHost.TemperatureSensorService日志类别（来自 ILogger<TemperatureSensorService>）
[0]事件 ID（EventId）
消息正文消息模板填上参数后的结果

9. 今天反复踩的坑

  • ILogger<T> 的 T 不是消息类型，是日志类别
  • 同一段顶层语句里 var services = ... 不能重复声明（笔记要写注释）
  • 字符串插值 vs 结构化日志：$"...{x}" 会丢失结构化字段
  • 结构化日志按位置匹配，写反顺序会数据串
  • Scoped ≠ per user，默认 per HTTP request
  • Action<int> 是类型，类型参数用 <>，不是 ()
  • lambda 参数 builder 不是“缺乏定义”，是 lambda 自身定义
  • lambda 没 return 不是错误，是 Action<T> 返回 void
  • 扩展方法的 this 写在定义里，调用时不写
  • dotnet build 必须在含 .csproj 的目录运行（今天遇到 csproj 误进 bin 的事故）

10. 一句话压缩

DI： 类声明依赖，容器创建依赖。
ILogger<T>： T = 日志类别；消息和字段来自 LogXxx(...) 的模板和参数。
Lifetime： 请求级 → Scoped；全应用共享 → Singleton；轻量无状态 → Transient。
Lambda： 没有名字的方法。参数由委托签名推断，Action<T> 不要 return。
<> vs ()： <> 给类型加类型参数；() 给方法/委托加调用参数。
扩展方法： this 写在定义里，调用时省略；编译期把点语法改写成静态方法调用。
