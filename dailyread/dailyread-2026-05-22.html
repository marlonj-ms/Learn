DI + Lambda 复盘日（无新代码）

2026-05-22 · 5 分钟复习 · 配套：完整 Summary

今天主线：不写新代码，纯口头复盘前一天的 DI + Lambda 心智模型。把"注册 ≠ 构造"和"生命周期 ≠ 调用次数"两条铁律说到嘴里能脱口而出。

1. 业务数据 vs 依赖项 —— 放哪里

角色放哪里例子
业务数据（business data）方法参数double celsius
依赖项（dependency）构造函数参数 → 私有只读字段ILogger<T> / IInventory
DI 不是"类不声明依赖"，而是 类清晰声明依赖，但不创建具体依赖。构造函数说"我需要什么"，容器说"我给你"。

2. 三个生命周期的精确表述

Lifetime规则同一 scope 内两次 resolve跨 scope
Transient每次 resolve 都 new不同对象不同对象
Singleton整个 provider 一个同一对象同一对象
Scoped同一 scope 一个同一对象不同对象
易混点：Lifetime 决定的是复用策略（reuse policy），不是调用次数。
"Transient 下 order1 ≠ order2"是因为我们 resolve 了两次，不是因为方法被调了两次。

3. DI 端到端流程图（背下来）

ServiceCollection (rules only — 注册清单)
  -> AddLogging / AddTransient / AddScoped / AddSingleton
  -> BuildServiceProvider()
  -> GetRequiredService<T>()
       -> 递归解析构造函数参数
       -> 自底向上 new 对象图
  -> 返回根服务

4. 接口映射 —— DI 的桥

// OrderService 依赖抽象 IInventory，不直接 new FakeInventory
public sealed class OrderService(ILogger<OrderService> logger, IInventory inventory)
{
    // ...
}

// 注册时必须给抽象映射到具体
services.AddTransient<OrderService>();
services.AddTransient<IInventory, FakeInventory>();   // ← 没这行 resolve 时炸
没注册映射 → 容器看到 IInventory 不知道 new 谁 → 运行时 InvalidOperationException: Unable to resolve service for type 'IInventory'。编译过，运行炸，这是 DI 最常见的坑。

5. Lambda 三件套复习

形式例子
表达式 lambda（expression lambda）x => x * 2
语句 lambda（statement lambda）x => { Console.WriteLine(x); return x * 2; }
0 参数() => "hi"（括号必填）
1 参数x => x * 2（括号可选）
多参数(a, b) => a + b（括号必填）

委托变量 vs 调用结果

Func<int> clock = () => DateTime.Now.Second;
clock      // ← 委托变量本身（一个值）
clock()    // ← 调用它，得到 int 结果

6. LINQ 延迟执行 —— ToList 是 trigger

var query = numbers.Where(n => n > 5);    // ❌ 还没跑！只构建了表达式
var list  = query.ToList();               // ✅ 这一刻才执行
foreach (var x in query) { ... }          // ✅ 这一刻也会执行
没 ToList() 时，每次 foreach 都会重新执行整个 LINQ 链。如果源是数据库或大集合，这是性能杀手。

7. new() target-typed 语法

// 老写法
public List<string> Items { get; } = new List<string>();

// C# 9+ target-typed new
public List<string> Items { get; } = new();   // ← 类型从左侧推断
注意：这不是 setter 赋值！get;-only 属性也能用 = new() 初始化，因为它初始化的是底层字段（backing field），发生在构造期。

8. 今天背下来的三句话

1. "Registering is not constructing; resolving is constructing."
2. "Lifetime controls reuse policy, not call count."
3. "Interface mapping is the bridge between abstraction and concrete implementation."

🎯 一句话总结

今天不写代码，把脑里的 DI 心智模型擦干净。注册阶段只是"写规则"，没有任何构造函数会跑；解析阶段才真正 new 对象。Lifetime 选哪个，看的是"这个对象什么时候该被复用"，不是"它会被调用几次"。
