Part A · DI 的 4 个阶段（用 trace 看到的）

阶段代码做了什么构造函数有没有跑
1. REGISTRATIONservices.Add...把 (ServiceType → ImplementationType, Lifetime) 塞进列表❌ 没有
2. BUILDBuildServiceProvider()把注册列表"冻结"成可解析的容器❌ 没有
3. RESOLVEGetRequiredService<T>()按需构造依赖图（bottom-up）✅ 这里才跑
4. USE业务调用调方法处理数据已经构造完，无关
锚句：Registration ≠ construction. BuildServiceProvider() only freezes the rule book.

Part B · Lifetime-Drill 的发现

public sealed class ProbeService
{
    public Guid Id { get; } = Guid.NewGuid();
    public ProbeService() => Console.WriteLine($"[CTOR] new Probe {Id}");
}
注册方式同 scope 两次 resolve跨 scope结论
AddTransient2 个 Guid2 个 Guid每次都 new
AddSingleton1 个 Guid1 个 Guidprovider 共享
AddScoped1 个 Guid2 个 Guidscope 内共享
反模式：把 Scoped 注入 Singleton → Singleton 永远持有第一次 scope 的对象 → 数据陈旧 / 内存泄漏。
ValidateScopes = true（ASP.NET Core 在 Development 环境自动开启）能在启动期抓出这种错。

Part C · "23 个 services" 之谜

调用 AddLogging(...) 之后 services.Count 跳到 23，不是 bug。
原因：AddLogging 是注册宏（registration macro），内部会偷偷追加：ILoggerFactory / ILogger<> / Filter 配置 / Options 管线（IOptions<>、IOptionsMonitor<>）等。
心智模型：每个 AddXyz 都是一个"导入 service descriptor 的包"。

Part D · API 的真定义

API = 调用方和被调用方之间的契约（contract）。
契约 = 签名（signature / shape） + 语义（semantic / behavior）。
锚句："Compiler checks the signature. Humans (and tests) check the semantics."
层级例子
方法 APIdouble RecordReading(double celsius)
类的公开 API类的 public 成员
库 APITemperatureSensor.Core.dll 的 public 表面
进程间 API（IPC）pipe / shared memory / unix socket
网络 APIHTTP / gRPC / GraphQL ← 明天目标

Part E · HTTP 比方法调用多一种结局

结局
Method call① 抛异常 ② 返回值（2 种）
HTTP call① 抛异常 ② 返回值 ③ 消息送达但响应丢了（3 种）
第三种结局叫部分失败（partial failure）：服务器记录成功 → 响应丢包 → 客户端以为失败 → 重试 → 同一笔被记两次。
所以 HTTP 世界必须懂：幂等性（idempotency）、重试 + 去重、Idempotency-Key 头。

Part G · 状态码三段心法 ⭐

"2xx 我做了。4xx 你错了。5xx 我错了。4xx 不重试，5xx 看幂等。"
范围含义常见客户端该重试吗？
2xx我做了200 / 201 / 202 / 204—
4xx你错了400 / 401 / 404 / 409❌
5xx我错了500 / 502 / 503✅（前提：幂等）

2xx 内部细分

  • 200 OK — 做完了，结果在响应体里
  • 201 Created — 已创建资源，新地址在 Location 头
  • 202 Accepted — 已收下，还在处理（异步）
  • 204 No Content — 做完了，没东西返回

Part H · REST = URL 名词 + HTTP 动词 + stateless

操作RESTful 写法非 RESTful 写法
读用户GET /users/42GET /getUser?id=42
创建用户POST /usersPOST /createUser
更新用户PUT /users/42POST /updateUser
删除用户DELETE /users/42POST /deleteUser?id=42
stateless（无状态）：每个请求自带所有信息，服务器不在内存里记上一次。这样多台服务器才能负载均衡，挂了一台不影响业务。
