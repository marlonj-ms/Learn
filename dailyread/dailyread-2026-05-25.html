Minimal Web API + IOptions 配置

2026-05-25 · 5 分钟复习 · 配套：完整 Summary · Program.cs

今天主线：从零 scaffold 一个 ASP.NET Core minimal API，把昨天写的 TemperatureSensor.Core 库挂上 HTTP 入口。然后层层叠加 3 个生产模式：结构化日志、启动期事件订阅、外部化配置。
结果：3 个 curl 探针验证通过：25°C → 200 / NaN → 400 / 100°C → 200（带 warn 日志）。

1. 7 步配方（locked-in）

1. Scaffold     dotnet new web -n TemperatureSensor.Api --framework net9.0
2. Wire deps    dotnet add reference ..\TemperatureSensor.Core\TemperatureSensor.Core.csproj
3. Register     builder.Services.AddSingleton<Sensor>(sp => new Sensor(threshold));
4. Build        var app = builder.Build();
5. Configure    app.MapGet("/", () => "Hello"); app.MapPost("/readings", ...);
6. Run          app.Run();
7. Test         curl -i -X POST http://localhost:5000/readings -H "Content-Type: application/json" -d '{"celsius":25}'
每一步停下来确认 → 再做下一步。"一步一确认"是新手在陌生生态里唯一能避免迷路的方法。

2. Minimal API 核心 4 个调用

调用作用
app.MapGet(path, handler)注册 GET 路由
app.MapPost(path, handler)注册 POST 路由
Results.Ok(value)返回 200 + JSON body
Results.BadRequest(value)返回 400 + JSON body

路由 handler 的参数 = DI 注入点

app.MapPost("/readings", (Sensor sensor, ILogger<Program> logger, ReadingDto dto) =>
{
    if (double.IsNaN(dto.Celsius) || double.IsInfinity(dto.Celsius))
        return Results.BadRequest(new { error = "invalid temperature" });

    sensor.RecordReading(dto.Celsius);
    logger.LogInformation("Recorded reading: {Celsius}°C", dto.Celsius);
    return Results.Ok(new { recorded = dto.Celsius });
});
谁是 DI 注入，谁是 body 绑定？框架按类型推断：
• 已注册的服务（Sensor / ILogger<T>）→ 从容器取
• 复杂对象 + 来自 POST → 反序列化 JSON body（ReadingDto dto）

3. 结构化日志（Structured Logging）

// ❌ 字符串拼接：日志只是文本，搜不出来
logger.LogInformation("Recorded reading: " + dto.Celsius + "°C");

// ✅ 命名占位符（named placeholder）：保留字段，能查询
logger.LogInformation("Recorded reading: {Celsius}°C", dto.Celsius);
{Celsius} 不是 string.Format！它在 App Insights / Seq / Datadog 里会以独立字段存在，可以 WHERE Celsius > 30 直接查询。这是生产可观测性的基础。

4. 组合根（Composition Root）— 启动期订阅

// ✅ 启动期订阅一次，长生命周期复用
var sensor = app.Services.GetRequiredService<Sensor>();
var startupLogger = app.Services.GetRequiredService<ILogger<Program>>();
sensor.ThresholdExceeded += (s, e) =>
    startupLogger.LogWarning("Threshold exceeded: {Celsius}°C", e.Celsius);

app.Run();
反模式（在路由 handler 内部 +=）： 
app.MapPost("/readings", (Sensor sensor, ILogger<Program> logger, ReadingDto dto) =>
{
    sensor.ThresholdExceeded += (s, e) => logger.LogWarning(...);   // ❌ 每次请求都新订阅
    sensor.RecordReading(dto.Celsius);
    return Results.Ok();
});
每次请求都新增一个 handler，4 次超温请求 = 1 + 2 + 3 + 4 = 10 条 warn 日志 → 内存泄漏 + 日志噪声。
多播委托按引用比较，即使 lambda 长得一样也不去重。

5. 外部化配置：appsettings.json + IConfiguration

// appsettings.json
{
  "Sensor": { "Threshold": 30.0 }
}

// Program.cs
double threshold = builder.Configuration.GetValue<double>("Sensor:Threshold");
key 写法来源
"Sensor:Threshold"appsettings.json 层级（: 是层级分隔符）
$env:Sensor__Threshold = "50"环境变量（双下划线 __ 等价 :）

配置提供者链（later wins）

appsettings.json
  -> appsettings.{Environment}.json   (开发环境覆盖生产)
  -> 环境变量
  -> 命令行参数                          ← 优先级最高
实测：设 $env:Sensor__Threshold = "50" 重启 → 启动日志显示 loaded from config: 50°C → POST 35°C 不报警（35 < 50）。同一份二进制，不同的阈值，不需要重新编译。这就是生产部署的玩法。

6. ILogger<T> 是怎么注入的（开放泛型）

框架注册的是开放泛型（open generic）：ServiceDescriptor.Singleton(typeof(ILogger<>), typeof(Logger<>))。
你 resolve 任何 ILogger<Foo>，容器在解析时把 <> 闭合成 <Foo>。一次注册，无限种 T。

7. 今天背下来的三句话

1. "路由 handler 的参数 = DI 注入点 + body 绑定点。框架按类型分。"
2. "启动期订阅一次，长寿命；请求里订阅每次叠加，泄漏。"
3. "配置链：appsettings → env-specific → env vars → CLI。Later wins。"
