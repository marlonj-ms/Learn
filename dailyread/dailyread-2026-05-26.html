Part A · 集成测试（Integration Tests）

1. Unit test vs Integration test 的真正边界

维度Unit TestIntegration Test
目标一个 class / 一个方法多个组件协作（routing + DI + JSON + 业务）
依赖都 mock 掉真实组件协作
入口直接调方法走 HTTP（HttpClient）
速度极快（毫秒）较慢（需要启动 host）
锚句：单元测试是显微镜，集成测试是风洞。两个都要有。

2. WebApplicationFactory<Program> — 内存里启动整个 API

public class TemperatureSensorAPITests : IClassFixture<WebApplicationFactory<Program>>
{
    private readonly HttpClient _client;

    public TemperatureSensorAPITests(WebApplicationFactory<Program> factory)
    {
        _client = factory.CreateClient();
    }
}
组件作用
WebApplicationFactory<TEntryPoint>在同一个进程跑你的 Program.cs，不绑真实端口
IClassFixture<T>xUnit 类级共享生命周期：一个测试类只 new 一次 factory
factory.CreateClient()拿到内存 HttpClient，直接 GetAsync("/") 就命中路由
省钱点：不开端口、不启动 Kestrel TCP listener，纯走 in-memory pipeline。比 dotnet run + 真 HTTP 快 10x+。

3. Program.cs 必须开"小窗户"

// Program.cs 末尾
public partial class Program { }   // ← 测试可见的"挂钩点"
顶级语句默认把 Program 类做成 internal，测试项目（另一个 assembly）拿不到。
没这一行 WebApplicationFactory<Program> 编译都过不去。这是微软官方推荐的 standard pattern。

4. EnsureSuccessStatusCode 是双刃剑

// ✅ 正路径
response.EnsureSuccessStatusCode();          // 非 2xx 就 throw

// ❌ 反路径不能用：会因为 400 抛出，反而让"我期望 400"的测试假失败
// ✅ 测错误路径永远直接断言 StatusCode：
Assert.Equal(HttpStatusCode.BadRequest, response.StatusCode);

5. PostAsJsonAsync — 不用手写 JSON

var reading = new { Celsius = 25.0 };          // 匿名对象
var response = await _client.PostAsJsonAsync("/readings", reading);
PostAsJsonAsync（来自 System.Net.Http.Json）做 3 件事：① 序列化对象 → JSON ② 设 Content-Type: application/json ③ 发 POST。
比手搓 StringContent + Encoding.UTF8 + "application/json" 干净 5 倍。

Part B · Docker 多阶段构建（Multi-Stage Build）

1. 为什么要多阶段？

单阶段（用 SDK 镜像）多阶段
体积~1.5 GB~92.9 MB（节省 94%）
包含什么SDK + 编译器 + NuGet 缓存 + 调试符号只 runtime + publish 产物
攻击面大（编译器是攻击者最爱）小

2. 完整 Dockerfile

# Stage 1: Build
FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build
WORKDIR /src
COPY TemperatureSensor.Api/TemperatureSensor.Api.csproj TemperatureSensor.Api/
COPY TemperatureSensor.Core/TemperatureSensor.Core.csproj TemperatureSensor.Core/
RUN dotnet restore TemperatureSensor.Api/TemperatureSensor.Api.csproj
COPY . .
RUN dotnet publish TemperatureSensor.Api/TemperatureSensor.Api.csproj -c Release -o /app/publish

# Stage 2: Runtime
FROM mcr.microsoft.com/dotnet/aspnet:9.0
WORKDIR /app
COPY --from=build /app/publish .
EXPOSE 8080
ENV ASPNETCORE_URLS=http://+:8080
ENTRYPOINT ["dotnet", "TemperatureSensor.Api.dll"]

3. 锁定比喻（背下来）

"FROM 是 stage 分界线" — 每个 FROM 启动一个全新的临时容器
"只有 COPY --from=<stage> 是桥" — ENV / WORKDIR / 安装的包都不跨 stage
"Stage 2 没编译器" — 想跑 dotnet build 都没工具，只能跑预编译好的 .dll
"92.9 MB 证明 Stage 2 是 runtime 不是 SDK" — 体积本身就是 multi-stage 正确性的证据

4. Dockerfile 必须放在共同父目录

2026-05-25/
├── Dockerfile               ← 这一层
├── TemperatureSensor.Api/
└── TemperatureSensor.Core/
锁定心法："Dockerfile 必须看得见所有它要 COPY 的项目。"
docker build . 的 . 是 build context 根。Dockerfile 里 COPY TemperatureSensor.Api/... 是相对 context 根的。必须把 Dockerfile 放在所有 COPY 项目的共同祖先（common ancestor）目录。

5. Layer cache 优化：先 csproj，后源码

# ✅ 高效
COPY *.csproj ./
RUN dotnet restore
COPY . .
RUN dotnet publish

# ❌ 低效（每次改 C# 都会重 restore）
COPY . .
RUN dotnet restore
RUN dotnet publish

Part C · 实战命令

# 构建镜像
docker build -t temperature-sensor:dev .

# 跑容器
docker run -d --name ts -p 8080:8080 temperature-sensor:dev

# 测试（注意：PowerShell 的 curl 是 Invoke-WebRequest 的别名！）
curl.exe -i http://localhost:8080/
curl.exe -i -X POST http://localhost:8080/readings -H "Content-Type: application/json" -d "{\"celsius\":25}"

# 清理
docker stop ts; docker rm ts
PS 的 curl 是别名陷阱：PowerShell 把 curl alias 到 Invoke-WebRequest，-X -d 都不认。必须显式写 curl.exe（Windows 10+ 自带）。JSON body 还要双重转义："{\"celsius\":25}"。
