Part C · 关键 yml 片段

1. 权限块（最小权限原则）

jobs:
  build-test-and-image:
    runs-on: ubuntu-latest
    permissions:
      contents: read         # ← 读源码
      packages: write        # ← 写 GHCR
GitHub Actions 给每个 job 自动注入的 GITHUB_TOKEN 默认只读。要写 GHCR / Issues / PR comments，必须显式 opt-in。

2. 登录 GHCR

- name: Log in to GHCR
  uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}

它内部其实就跑：echo $TOKEN | docker login ghcr.io -u marlonj-ms --password-stdin。

3. build + push 一步

- name: Build & push image
  uses: docker/build-push-action@v6
  with:
    context: 2026-05-25
    push: ${{ github.event_name == 'push' }}
    tags: |
      ghcr.io/marlonj-ms/temperature-sensor:${{ github.sha }}
      ghcr.io/marlonj-ms/temperature-sensor:latest
多 tag 不重新构建。同一个 image id 贴两个名字而已，latest 和 sha 永远指向同一个二进制。

Part D · GitHub Actions context 变量（要记的）

变量值用途
${{ github.sha }}触发本次 run 的 commit SHA给镜像打不可变 tag
${{ github.actor }}触发本次 run 的 GitHub 用户名docker login 的 username
${{ github.event_name }}push / pull_request / schedule分支判断（PR 不推、只 main 推）
${{ secrets.GITHUB_TOKEN }}自动注入的 job-scoped token不用自己生成 PAT
陷阱：${{ github.sha }} 是触发 run 的 commit 的 sha，不是 yml 文件本身的 sha。改 yml 文件 push 一次，那次的 commit sha 就是 github.sha。

Part E · GHCR 推成功了为什么 REST 返 401？

原因：包默认是 private。未认证的 REST endpoint 看不到私有包 → 返 401 Unauthorized。这不代表 push 失败。

3 种查看方式：

方式命令 / 链接
1. UI（最快）https://github.com/marlonj-ms?tab=packages
2. 认证 RESTInvoke-RestMethod -Headers @{Authorization='Bearer <PAT>'} -Uri 'https://api.github.com/users/marlonj-ms/packages/container/temperature-sensor/versions'
3. docker pulldocker login ghcr.io -u marlonj-ms（密码用 PAT, read:packages scope）→ docker pull ghcr.io/marlonj-ms/temperature-sensor:latest

� 晚场补充：Kestrel + SDK + 反向代理 概念深挖

早上 CI 收尾后，从一个 Dockerfile 的 SDK/runtime 问题出发，把 Kestrel 真身、反向代理、容器网络 namespace、云原生扩缩容串起来。配套：完整 Summary

1. 三层镜像金字塔

mcr.microsoft.com/dotnet/sdk:9.0       ~850 MB  ← 含编译器 + ASP.NET Core 框架 + Runtime
mcr.microsoft.com/dotnet/aspnet:9.0    ~220 MB  ← 含 ASP.NET Core 框架 + Runtime
mcr.microsoft.com/dotnet/runtime:9.0   ~200 MB  ← 只有 .NET Runtime（跑 web app 会启动失败）
多阶段配方：Stage 1 (build) 用 sdk 编译 → Stage 2 (run) 用 aspnet 跑。
为什么 web app 不能用 runtime？ASP.NET Core 是独立的 shared framework，runtimeconfig.json 启动期检查发现没装就 framework not found 立刻失败。

2. Kestrel 是真 web server（不是测试工具）

误区："Kestrel 是测试的虚拟 IIS 替身，生产要换成真 web server。"
正解：Kestrel 在生产里就是直接接 TCP、解析 HTTP、跑你 C# 代码的那个家伙。没有任何东西替代它。
层角色类比
客户端发请求顾客
反向代理（Envoy / nginx / IIS-as-proxy）TLS termination、负载均衡、缓存、边缘安全餐厅前台
Kestrel解析 HTTP、调你的 endpoint handler厨师（永远是它做菜）
你的 C# 代码业务逻辑菜谱

3. 10 个容器都听 8080，怎么不打架？

关键概念：Linux 给每个容器独立的 网络命名空间（network namespace）。
容器 A 的 8080 跟容器 B 的 8080 根本不是同一个东西，在两个隔离世界里。
冲突只发生在你想从容器外面访问时 — 那时候靠 docker run -p 5001:8080 在 host 上做端口映射。 
┌─ Container 1 ──┐  ┌─ Container 2 ──┐  ┌─ Container 10 ─┐
│ Kestrel :8080  │  │ Kestrel :8080  │  │ Kestrel :8080  │  ← 内部端口，各自独立
└────────────────┘  └────────────────┘  └────────────────┘
        ↑                    ↑                    ↑
        └────────────────────┴────────────────────┘
                             │
                  ┌──────────┴─────────┐
                  │  反向代理 / LB     │  ← 这一层做分发
                  └────────────────────┘

4. 反直觉：Kestrel 比 IIS 更轻更快

指标IIS + WorkerKestrel（同进程）
进程数2 个1 个
请求路径client → IIS → IPC → workerclient → Kestrel（无 IPC）
跨平台Windows onlyWin / Linux / macOS
TechEmpower benchmark中等全球前 5
启动时间慢~1 秒

为什么反而轻？异步 I/O 模型（async/await）+ 同进程无 IPC + Span<T>/ArrayPool<T> buffer 池减少 GC。

5. 云原生设计的精髓：代码层稳定，元数据层动态

ACA 从 3 replica 扩到 10 replica 时：
✅ 变：Envoy 路由表里登记的后端 IP 列表（要加新机器）
🚫 不变：host 上的 443、容器内部 8080、你的 C# 代码

范式翻转：以前靠"把 server 做大"扛流量；现在靠"把 app 做小、做多、做独立"扛流量。

6. 晚场新加的坑（cheatsheet 35–37）

坑 35：runtime:9.0 没有 ASP.NET Core 框架 — web app 用它跑会启动立刻失败，framework not found。要用 aspnet:9.0。
坑 36：容器内部端口 ≠ host 端口 — 10 个容器都用 8080 在容器内合法；本地多开必须改 host 端口（-p 5001:8080）。
坑 37：反向代理 ≠ Kestrel 的替代品 — Envoy/nginx/IIS-as-proxy 都在 Kestrel 前面，Kestrel 永远还在跑。

3. Revision 是 immutable — 云原生回滚的物理基础

az containerapp update --image .../v2
    ↓
ACA 不修改旧 revision，而是：
  1. 创建新 revision: temperature-sensor--abc123
  2. 等新 rev 健康 → 切 100% traffic
  3. 旧 rev 保留 → 秒级回滚兜底

反例：传统 IIS 直接覆盖 wwwroot，出问题只能祈祷有备份。

4. Pod 数 = 你真正在花的钱

min / max0 流量时流量爆炸时账单

5. Autoscaler 不是即时的

请求来了
   ↓
观察窗口 ~30s
   ↓
评估指标超阈值 → 决定扩
   ↓
启动新 pod ~10-20s
   ↓
~50 秒后才看得见新 pod
Cooldown：scale-out 后 ~5 分钟才会开始 scale-in（防抖动）。
默认 HTTP 规则：每个 replica 同时 10 个并发才扩。30 个 curl 串行返回太快 → 根本没堆积到阈值。
结论：想看扩缩演示要么持续 60s+ 真并发，要么直接 --min-replicas 2。

6. PowerShell 并发 — 5.1 vs 7+ 的坑

方式隔离启动开销版本
Start-Job进程（fork pwsh）慢 ~1sPS 5.1+
Start-ThreadJob线程（同进程）快 ~50msPS 5.1+（自带）
ForEach-Object -Parallelrunspace中PS 7+ only
今天踩的坑：ForEach-Object -Parallel 在 Windows PS 5.1 报 AmbiguousParameterSet。
5.1 根本没有这个特性。要么升 pwsh 7+，要么换 Start-ThreadJob / Start-Process。

3. OIDC 7 步握手——为什么不需要 client-secret

setup（1 次性）：
  你在 Azure 配 federated credential：
    “我信任 GitHub OIDC issuer，且 sub 必须 = repo:marlonj-ms/Learn:ref:refs/heads/main”

runtime（每次 push）：
  1. runner 启动 azure/login
  2. runner 找 GitHub OIDC service 要 JWT
  3. GitHub 用私钥签个 JWT
       sub = "repo:marlonj-ms/Learn:ref:refs/heads/main"
       exp = 10 分钟后
  4. runner 把 JWT 递给 Azure
  5. Azure 拉 GitHub 公钥 → 验签 → 查名单 → 发 access token
  6. runner 用 access token 跑 az containerapp update
锒句：“Azure 不存 GitHub 的 secret。Azure 只信 GitHub 的公钥。没有长期密钥，没有泄露面。”

Part A · 第一个锚点：接口 ≠ DI

用户读到 public interface IAuditable { ... } 第一反应："这是 DI 吗？"——这是个关键混淆，今天先把它彻底劈开：

角度接口（interface）DI 容器（IServiceCollection）

1. 类的 5 个部位一次对位

public class Order : IAuditable                      // ④ 继承/接口实现
{
    public required string Id { get; init; }         // ③ 属性 + init + required
    public DateTime CreatedAt { get; init; }

    public decimal Amount { get; init; }

    public override string ToString()                // ② 方法
        => $"Order(Id={Id}, Amount={Amount})";       //    用 => 简写
}

Order o = new() { Id = "A-100", Amount = 88.8m } // (3) target-typed (C# 9+)
var list = new List<int> { 1, 2, 3 };            // (4) 集合初始化器
要点：(2) 不是"括号被吞了"，而是 C# 允许在 { } 跟随时省略无参括号。底层依然走无参构造 + 属性赋值。

4. required × 默认值 — 默认永远不会赢

public required string Id { get; init; } = "DEFAULT";
//          ↑ 编译期门卫              ↑ 运行期默认值

new Order { /* Id 没传 */ }   // ❌ CS9035 — 编译失败
new Order { Id = "DEFAULT" }  // ✅ 必须显式传
锚句："required 是编译期门卫，默认值是运行期值。编译失败的代码永远走不到运行期，所以默认值在 required 面前永远落败。"

Part C · 泛型约束 + 一个 CS 错误

1. new() 约束是关于"方法体"，不是关于 T

public class Repository<T> where T : class, IAuditable, new()
{
    public T CreateDefault() => new T();   // ← new() 约束存在的全部原因
}

不写 new()，编译器不允许你在方法体里写 new T()——因为它没法保证 T 有公开无参构造。

2. CS9040 — required 和 new() 互斥

public class Order : IAuditable
{
    public required string Id { get; init; }
}

public class Repository<T> where T : class, IAuditable, new()
{
    public T CreateDefault() => new T();   // ❌ CS9040
    //  T = Order，Order 有 required 成员 → 不能无参构造
}
解法：把 new() 约束 + CreateDefault() 一起注释掉。required 已经把"必须传值"写进类型签名；new() 承诺"我能无参构造"——两个承诺直接打架，编译器替你提前发现。

3. 常见泛型约束矩阵

约束含义
where T : classT 必须是引用类型
where T : structT 必须是值类型
where T : SomeBaseT 必须继承自 SomeBase
where T : IInterfaceT 必须实现接口
where T : new()T 必须有公开无参构造
where T : notnullT 不可为 null

Part D · Variance — 表层进入 + 显式 PARK 🅿️

知道的部分（表层）

  • out T = 协变 (covariance)，允许 IProducer<string> 当 IProducer<object> 用。
  • in T = 逆变 (contravariance)，允许 IConsumer<object> 当 IConsumer<string> 用。
  • 没标 out/in = 不变 (invariance)，C# 默认。
  • 实验验证过：删 out → exit 1；恢复 out → exit 0 ✅

Park 的部分

"为什么 out 就能让协变安全"——这层深度 reasoning 今天没拿下。用户主动喊停"这里暂时放弃，回头重新学"——mentor 即刻 honor。知道何时该 park，比硬怼一个不明白的概念更重要。
