Join + SelectMany + Yield + IQueryable 🔘

2026-05-29 (PM · Day 52) · Tier-1 LINQ 14 个收官 · 配套：完整 Summary · Join demo · SelectMany demo

一句话：「LINQ 的真相不是『方法』，是『方法构造一个状态机 / Expression Tree，有人按按钮才跑』。Join 配 hash 表 O(N+M)，SelectMany 拍扁嵌套；但所有链子背后是同一条线 —— 谁按按钮决定何时执行，左边静态类型决定走 Enumerable.Where 还是 Queryable.Where，返回类型决定下游能不能继续推 SQL。」

Part 1 · Join — 5 args / 4 generics / hash join O(N+M)

customers.Join(
    orders,
    c => c.Id,                                  // 左键
    o => o.CustomerId,                          // 右键
    (c, o) => new { c.Name, o.OrderId, o.Amount });
// = SQL INNER JOIN，键不匹配两边都丢
  • 4 个泛型：TOuter / TInner / TKey / TResult
  • 内表先 hash 进字典，外表再扫一遍按键查 → O(N+M)

Part 2 · SelectMany — 拍扁 + 父子展开

形态用途
.SelectMany(arr => arr)1-arg：纯拍扁 [[1,2],[3]] → [1,2,3]
.SelectMany(u => u.Roles, (u, r) => new { u.Name, Role = r })2-arg：父子展开成扁平表（ORM 一对多神器）

Part 3 · 🪤 CS0579 — 同目录两个 csproj

SDK 默认行为：当前目录所有 *.cs 都自动 include。两个 csproj 同目录 → 两份 .cs 都被 include 两遍 → 两次顶级语句 + 两套 assembly 属性 → CS0579: Duplicate Attribute。

修复：每个 demo 各自一个子文件夹（口诀 #68）

Part 4 · yield + 状态机 —— LINQ 的真正引擎

static IEnumerable<int> Demo()
{
    Console.WriteLine("A");
    yield return 1;
    Console.WriteLine("B");
    yield return 2;
}

// 编译器实际生成：
private sealed class <Demo>d__0 : IEnumerable<int>, IEnumerator<int>
{
    private int _state;
    public bool MoveNext()       // ← 真正的"按按钮"入口
    {
        switch (_state)
        {
            case 0: Console.WriteLine("A"); _current = 1; _state = 1; return true;
            case 1: Console.WriteLine("B"); _current = 2; _state = 2; return true;
            case 2: return false;
        }
    }
}
口诀 #70：var seq = Demo(); 只搭管子，body 一行都不跑。foreach / ToList 才把水放进管子。

Part 5 · 谁是按按钮的人 🔘

按按钮？操作
✅foreach, ToList, ToArray, ToDictionary, Count, First, Single, Any, All, Sum, Max, string.Join, await foreach
❌Where, Select, SelectMany, Join, OrderBy, GroupBy, Take, Skip —— 只是再套一层状态机

Part 6 · IQueryable vs IEnumerable —— 同 lambda，两个世界

维度IEnumerable<T>IQueryable<T>
lambda 编译成Func<T,bool>Expression<Func<T,bool>> = 表达式树
.Where 实际调用Enumerable.WhereQueryable.Where
执行位置C# 进程内翻译成 SQL，DB 跑
典型来源List<T>EF Core DbSet<T>
lazy？✅✅
选哪条路 = 左边静态类型决定。 DbSet<T> : IQueryable<T>, IEnumerable<T>，两个都有，IQueryable 更具体 → 编译器优先挑它。所以 lambda 被装进表达式树。

💣 经典生产事故

// ❌ 凶手是 .ToList() 出现在 .Where() 之前
var orders = db.Orders.ToList();              // SELECT * 拉走 10M 行！
return orders.Where(...).Take(50).ToList();   // 已退化成内存过滤

// ✅ 让 SQL 干活
return db.Orders.Where(...).Take(50).ToList();

🪤 Repository 出门陷阱

public IEnumerable<Order> GetBigOrders(DbContext db)   // ← 返回类型砍 SQL 能力
//     ^^^^^^^^^^^^^^^^^^
{
    return db.Orders.Where(o => o.Amount > 10000).Take(50);
    // 内部仍是 IQueryable<Order>，出门时被向上转型为 IEnumerable
}

// 调用方
var bigRecent = repo.GetBigOrders(db)
                    .Where(o => o.Date > DateTime.Today)  // ← Enumerable.Where, 内存
                    .ToList();
// 50 行先回内存，再筛日期 —— 永远没机会让 DB 一次性过滤
口诀 #72：Repository 返回类型 = 契约。返回 IEnumerable<T> 等于挂"到此为止"的牌子；返回 IQueryable<T> 等于"你想叠条件随便"。

口诀 — 6 个新条目（#68–#73）

#口诀
68同级文件夹只能有 1 个 .csproj。否则 SDK 把所有 .cs include 进每个项目 → CS0579 重复属性。一 demo 一文件夹。
69IEnumerable<T> 是接口契约（"我能被 foreach"），运行时对象是 Iterator 类（WhereEnumerableIterator、<Demo>d__0 等）。能 foreach ≠ 是 List。
70状态机被动 —— 没人按按钮就不动。foreach/ToList/Count/First 才按按钮；LINQ 链子（Where/Select/...）不按按钮，只是再套一层。
71IEnumerable<T> = 本地过滤（filter here，lambda 是 Func）；IQueryable<T> = 远程过滤（filter there，lambda 是 Expression Tree → SQL）。
72Repository 方法的返回类型 = 契约。返 IEnumerable<T> 切断下游推 SQL 的能力；返 IQueryable<T> 留给下游继续叠条件。
73类型决定"翻译成什么"，按按钮决定"什么时候执行"。两件事都要满足。IEnumerable 和 IQueryable 都是 lazy 的。
